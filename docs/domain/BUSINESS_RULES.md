# Business Rules

**Purpose:** Every rule that governs the entities in [DOMAIN_MODEL.md](./DOMAIN_MODEL.md) — and, for each, whether it's **universal code** (true for every school) or **per-school configuration** (varies by state, board, or policy), per [CONFIGURATION_GUIDE.md](../CONFIGURATION_GUIDE.md)'s own framework. This split is the single most important thing this document gets right — treating a Rajasthan-specific or CBSE-specific value as universal code would silently break the model for every other school this platform is meant to serve.

---

## 1. Academic Year

| Rule                                                                                       | Type                                                                  |
| ------------------------------------------------------------------------------------------ | --------------------------------------------------------------------- |
| Exactly one `AcademicYear` per school has `isCurrent = true` at any time                   | Code (invariant, enforced at the mutation that sets it)               |
| An `AcademicYear`'s `endDate` must be after its `startDate`                                | Code                                                                  |
| The academic year's start month (April is the overwhelming Indian norm, but not universal) | Configuration (`AcademicYear.startDate` itself; no hardcoded "April") |
| Term structure — how many `ExamTerm`s, their names, their weight toward promotion          | Configuration (`ExamTerm` rows, `promotionPolicy` JSON)               |

## 2. Admission Eligibility

| Rule                                                                                                                                                            | Type                                                                                                                                                                                                                                                       |
| --------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| A student must meet the minimum age for the class applied for, as of a school-specified cutoff date                                                             | Code (the check) + Configuration (the actual age-per-class table and cutoff date)                                                                                                                                                                          |
| Minimum age values themselves (e.g., "Nursery: 3+ years as of March 31")                                                                                        | Configuration — **varies by state and has been actively litigated/revised**; never hardcode a specific number as if it were universal                                                                                                                      |
| No formal entrance test/interview below a school-configurable class threshold                                                                                   | Code (the restriction exists as a business rule) + Configuration (which class it applies below) — several states restrict or ban formal interviews for pre-primary/Class 1 admission; modeled as a policy toggle, not assumed present or absent everywhere |
| An "entrance interaction" (informal, age-appropriate) is distinct from an "entrance exam" and should never be labeled or scored as the latter for young classes | Code — a naming/UX discipline as much as a data rule, worth stating explicitly so a future implementation doesn't accidentally build a scored "Nursery exam"                                                                                               |

## 3. Admission Categories & RTE Quota

| Rule                                                                                                                                                 | Type                                                                                                                                                                                               |
| ---------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| The set of valid `admissionCategory` values (General, RTE, Management Quota, Staff Ward, Sibling, ...)                                               | Configuration — genuinely varies by school policy                                                                                                                                                  |
| RTE Section 12(1)(c) reserves 25% of entry-level seats in private unaided schools for Economically Weaker Section (EWS)/Disadvantaged Group children | Code (the concept exists, `RteDetails` entity) + Configuration (the actual 25% figure and which class counts as "entry-level" for this school — typically Nursery or Class 1, but school-specific) |
| An `AdmissionApplication` with `admissionCategory = RTE` must have an associated `RteDetails` record before it can move to `CONFIRMED`               | Code (referential/workflow invariant)                                                                                                                                                              |
| Government reimbursement tracking for RTE seats                                                                                                      | Future — likely Fee-module territory, not V1; `RteDetails.reimbursementStatus` exists as a placeholder field only, not a full workflow                                                             |

## 4. Attendance

| Rule                                                                                                                                                                                                          | Type                                                                                                                                                                      |
| ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| An `AttendanceSession` is unique per `(section, date)` — a section's attendance can be taken once per day, then edited, not re-created                                                                        | Code                                                                                                                                                                      |
| An `AttendanceRecord` must reference an `Enrollment` that is active for the `AttendanceSession`'s `academicYear` — a student cannot be marked present/absent for a section they weren't enrolled in that year | Code                                                                                                                                                                      |
| Attendance granularity is daily (one status per student per day), not period-wise                                                                                                                             | Code (V1 scope decision, matching [PRD § 6.2](../PRODUCT_REQUIREMENTS.md#62-teacher--daily-attendance)) — period-wise is a named-but-undesigned future variant, not built |
| Attendance percentage = `daysPresent / workingDays` for a given period, where `HALF_DAY` counts as configurable (commonly 0.5, some schools count differently)                                                | Code (the formula shape) + Configuration (how `HALF_DAY` weights)                                                                                                         |
| Only the assigned section teacher (or Admin) may mark/edit attendance for a section                                                                                                                           | Code — see [PERMISSION_MATRIX.md](./PERMISSION_MATRIX.md)                                                                                                                 |

## 5. Examinations & Grading

| Rule                                                                                                                                        | Type                                                                                                                                                                                                                                                        |
| ------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `MarksRecord.marksObtained` must be `>= 0` and `<= ExamSubjectSchedule.maxMarks`                                                            | Code — directly implements [PRD's acceptance criteria](../PRODUCT_REQUIREMENTS.md#7-acceptance-criteria-representative-examples): "when they enter a value exceeding the maximum marks... the system rejects the entry with a clear error"                  |
| `ExamSubjectSchedule.passMarks <= maxMarks`                                                                                                 | Code                                                                                                                                                                                                                                                        |
| A student's `grade` on a `MarksRecord` is derived from the school's active `GradeScale`, if one is configured                               | Code (the derivation) + Configuration (the actual `GradeScale.bands`)                                                                                                                                                                                       |
| No specific grading scale (CBSE 9-point, percentage-only, descriptive Nursery grades) is assumed universal                                  | Configuration — a school may run multiple `GradeScale`s simultaneously (e.g., descriptive for Nursery–2, marks-based from Class 3), selected per class, not globally fixed                                                                                  |
| Board affiliation does not drive exam/grading mechanics for this school's age range (Nursery–Class 8)                                       | Code (a documented non-rule) — board-conducted exams begin at Class 9/10 for essentially every Indian board; `Examination`/`GradeScale` are fully school-defined regardless of `School.affiliationBoard`, see [README.md § 4](./README.md#4-research-basis) |
| Only the teacher assigned to a subject/section (or Admin) may enter/edit marks for it                                                       | Code — see [PERMISSION_MATRIX.md](./PERMISSION_MATRIX.md)                                                                                                                                                                                                   |
| A `ReportCard`, once generated, is a frozen snapshot — later `MarksRecord` corrections do not retroactively alter an already-generated card | Code — regeneration creates a new dated `ReportCard`, it does not mutate the old one                                                                                                                                                                        |

## 6. Promotion & Detention Policy

This is the rule most likely to be wrong if hardcoded — worth stating the regulatory background explicitly, not just the rule:

- The **RTE Act 2009**, as originally enacted, established a no-detention policy through Class 8 nationally.
- The **RTE Amendment Act 2019** gave individual states the option to reintroduce examinations and a detention/re-examination policy for Class 5 and Class 8. **Rajasthan is one of the states that has exercised this option.** Other states have not. This is real, active regulatory divergence — not a hypothetical.

| Rule                                                                                                                                                                                         | Type                                                                                                                                                                               |
| -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Whether a school's state applies no-detention or exam-based promotion/detention, and for which classes                                                                                       | Configuration — `AcademicYear.promotionPolicy` JSON, per school, never hardcoded to either national default or Rajasthan's specific choice                                         |
| If exam-based: a student who fails is offered a re-examination before a final detention decision (per RTE Amendment Act 2019's own re-test provision, where the state has adopted detention) | Code (the re-exam workflow step exists) + Configuration (whether it applies, and the retest window)                                                                                |
| A `PromotionRecord.basis` must always be recorded ("ExamResult" / "NoDetentionPolicy" / "ReExam") — never a bare status with no rationale                                                    | Code — this is precisely why `PromotionRecord` is a first-class entity, not just a status flag on `Enrollment` (see [DOMAIN_MODEL.md § 6.3](./DOMAIN_MODEL.md#63-promotionrecord)) |
| A `PromotionRecord` with `outcome = PROMOTED` must reference a `resultingEnrollment` in the next `AcademicYear`                                                                              | Code                                                                                                                                                                               |

## 7. Transfer & Transfer Certificate

| Rule                                                                                                                                                                                        | Type                                                                                                                                                                          |
| ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| A `TransferCertificate` may only be issued for a `Student` whose final `Enrollment` has ended (withdrawal or transfer-out, not promotion)                                                   | Code                                                                                                                                                                          |
| The TC field set (admission number, category, attendance, conduct, promotion status, reason for leaving, fee dues) is treated as a near-universal required superset                         | Code (the entity's field list) — the exact _print layout_ is state-prescribed and varies; the underlying data captured does not, in practice, vary as much as the format does |
| A TC, once issued, is immutable; a correction issues a new dated TC referencing the same student, never edits the original                                                                  | Code — legal-record integrity                                                                                                                                                 |
| `TransferCertificate.udisePen` (via the linked `Student`) should be present before issuance where the government PEN system is in use, since a receiving school needs it to link continuity | Code (a should-warn, not necessarily a hard block — a student admitted before UDISE assignment or from an unrecognized prior school may legitimately lack one)                |

## 8. Data Protection (DPDP Act 2023)

Named here because it is real, applicable regulation, not because this task authorizes building a compliance engine:

| Rule                                                                                                                                                                                                          | Type                                                                                                                               |
| ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| Every `Student` field is, by default, a minor's personal data and subject to DPDP Act 2023 § 9 (verifiable parental consent for processing; no behavioral tracking/targeted advertising directed at children) | Code-level design principle — see [DATA_DICTIONARY.md § PII Classification Legend](./DATA_DICTIONARY.md#pii-classification-legend) |
| Guardian consent capture (who consented, when, to what) is a **future** requirement, not designed here                                                                                                        | Future — flagged as a real gap Epic B's actual implementation should scope explicitly, not silently skip                           |
| Analytics/reporting (see [REPORTING_MODEL.md](./REPORTING_MODEL.md)) must aggregate, not expose, individual minors' data by default                                                                           | Code-level design principle for any future dashboard                                                                               |

## 9. Teacher Assignment & Compliance Ratios

| Rule                                                                                                                                                                       | Type                                                                                                                                                                                                                                                                               |
| -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| A `TeacherAssignment` with `isClassTeacher = true` is unique per `Section` per `AcademicYear` — exactly one Class Teacher                                                  | Code                                                                                                                                                                                                                                                                               |
| A `Teacher` may hold multiple subject assignments across multiple sections simultaneously                                                                                  | Code (no artificial cap)                                                                                                                                                                                                                                                           |
| Pupil-Teacher Ratio (PTR) — RTE prescribes norms (commonly cited as 30:1 for primary, 35:1 for upper primary) that recognized schools should meet                          | Configuration/Reporting — not an enforced hard constraint on `TeacherAssignment` creation (a school may temporarily exceed it and still need to function), but a **reportable compliance metric** — see [REPORTING_MODEL.md § 5](./REPORTING_MODEL.md#5-future-compliance-reports) |
| Minimum teacher qualification norms (RTE-prescribed, e.g., TET pass) are a compliance fact tracked via `TeacherQualification`, not a hard gate preventing account creation | Configuration/Reporting — same reasoning as PTR: visibility, not a blocking constraint, since a school's real staffing reality shouldn't be blocked by the software                                                                                                                |

## 10. Summary — Configuration vs. Code, at a Glance

Per [CONFIGURATION_GUIDE.md § The Core Question](../CONFIGURATION_GUIDE.md#the-core-question), applied to this domain specifically:

| Always Code (universal)                                             | Always Configuration (per-school/per-state)                                                   |
| ------------------------------------------------------------------- | --------------------------------------------------------------------------------------------- |
| Marks cannot exceed max marks                                       | The max marks themselves, per subject/exam                                                    |
| Exactly one current academic year                                   | The academic year's start month and term structure                                            |
| A TC requires an ended enrollment                                   | The TC's exact print layout/format                                                            |
| Attendance session is unique per section per day                    | How `HALF_DAY` weights in the percentage formula                                              |
| A promotion record must state its basis                             | Whether the school's state uses no-detention or exam-based promotion, and for which classes   |
| An RTE application needs `RteDetails`                               | The admission category list itself; the RTE quota percentage and which class is "entry-level" |
| Grade is derived from the active grade scale                        | The grade scale's actual bands — and whether one exists at all                                |
| Board affiliation does not drive exam mechanics for Nursery–Class 8 | The board affiliation value itself (display/recognition purposes)                             |
