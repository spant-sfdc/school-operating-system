# Event Model

**Purpose:** The domain events that occur as [WORKFLOWS.md](./WORKFLOWS.md)'s processes run, documented so future consumers — notifications, the audit trail, reporting, and eventually a Parent Portal — have a stable vocabulary to build against. **No event bus, queue, or pub/sub mechanism is implemented or being proposed here.** In V1, every "event" below is realized simply as a direct database write plus an [`AuditLog`](./DOMAIN_MODEL.md#91-auditlog) row inside the same request — this document names the events conceptually so that if/when a real async mechanism (e.g., for SMS/notification integrations, per [PRODUCT_VISION.md § 9](../PRODUCT_VISION.md#9-future-expansion)) is ever justified, the vocabulary already exists rather than being invented under pressure.

## 2. Design Notes

- **Every event above is currently realized synchronously**, inside the same server-side mutation that causes it — e.g., `AttendanceSessionSubmitted` is not a message published to a queue, it's simply "the `AttendanceRecord` upserts completed and an `AuditLog` row was written." This document names them as events for _vocabulary and future-proofing_ reasons, not because an event-driven architecture is being introduced now — that would be exactly the kind of premature infrastructure [PRODUCT_VISION.md § 7](../PRODUCT_VISION.md#7-product-philosophy) warns against building ahead of real need.
- **`AuditLog` is the one universal consumer.** Every event that represents a state change writes an `AuditLog` row — this is the mechanism, not a queue subscription.
- **Notification consumers are uniformly future.** No SMS/email/push integration exists today; [PRODUCT_VISION.md § 9](../PRODUCT_VISION.md#9-future-expansion) already lists "SMS/notification providers" as a future integration, consistent with marking every notification-consumer row above as future, not a V1 commitment.
- **Naming convention**, for whenever a real implementation needs it: `{Entity}{PastTenseVerb}` — matches common domain-event naming practice, chosen so a future engineer building an actual event system doesn't have to invent a convention from scratch.

## 3. What This Enables Later (Not Now)

If a future task justifies building an actual async mechanism — most plausibly triggered by the Parent Portal epic ([PRODUCT_ARCHITECTURE.md § 6](../PRODUCT_ARCHITECTURE.md#6-parent-portal)) needing to fan out notifications — this catalog is the starting contract for event payloads. That task would still need its own scoped decision (transport mechanism: a simple job queue vs. a message broker; delivery guarantees; retry policy) — none of which is decided here.
