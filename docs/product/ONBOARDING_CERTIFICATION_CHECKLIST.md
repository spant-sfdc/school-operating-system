# Onboarding Certification Checklist

**Purpose:** The reusable checklist a TechPulse engineer follows to confirm a school can actually be onboarded using only this repository and its documentation — no chat history, no asking another engineer. Produced by [Sprint C2 — Client Onboarding Certification](../DECISIONS.md#d-042--sprint-c2-client-onboarding-certification), which exercised every item below live against a real (temporarily reset-to-fresh-state) deployment, not just read it from documentation. **Run this checklist before certifying any future client's onboarding as complete**, including the next real client, not only Pant Public School.

**Relationship to other checklists:** [GO_LIVE_CHECKLIST.md](./GO_LIVE_CHECKLIST.md) (product) gates the broader go-live decision (infrastructure, imported data, training) — this document gates one narrower, earlier question: **can the framework itself be stood up and configured at all, before any real client data enters the picture.** [docs/onboarding/GO_LIVE_CHECKLIST.md](../onboarding/GO_LIVE_CHECKLIST.md) gates public-website content readiness specifically. All three are independent gates; passing this one is a precondition for the other two, not a replacement.

## 4. Certification Findings — Fixed This Sprint

- **README.md was stale enough to actively mislead a new engineer.** Its Quick Start never mentioned running migrations or the seed script, described the project as still at "Milestone 4 — Admissions Experience" (many sprints out of date), and never linked to `docs/development/DEVELOPMENT_LOGIN.md` — the one document that actually names the Bootstrap Administrator's credentials. **Fixed**: Quick Start now states the full proven sequence (§ 1–2 above); Status and Roadmap sections rewritten to the current Epic-based state; Documentation Map now links `DEVELOPMENT_LOGIN.md` and `docs/product/README.md`.
- **`prisma/seed.ts` had no way to seed reference-data-only.** `docs/product/CLIENT_IMPLEMENTATION_PLAYBOOK.md § 2.1` step 5 and `docs/product/GO_LIVE_CHECKLIST.md § 2` both already _required_ a real client deployment to skip the generic Nursery–8 sample data, but no mechanism existed to do that — a new engineer would have had to manually diff and hand-edit the seed script, undocumented. **Fixed**: `SEED_DEV_FIXTURES=false` env flag, additive and backward-compatible (unset behaves exactly as before), proven live both ways this sprint.
- **`CLIENT_CUSTOMIZATION_GUIDE.md`'s own checklist item 11 was aspirational**, describing a manual code edit that was never actually needed to exist as "the" answer. **Fixed**: now cites the working `SEED_DEV_FIXTURES` flag directly.

## 5. Verified Non-Issues (Investigated, Not Assumed)

- **`"seed-school"` hardcoded literal** (`School.schoolId` in `prisma/seed.ts`) — confirmed via `grep -rn '"seed-school"' src/ prisma/` that it appears nowhere else in the codebase. Nothing compares against, displays, or assumes this value. Not a coupling point; a new client repository does not need to change it.
- **No duplicate setup/configuration/readiness logic** — confirmed via grep that `checkSystemReadiness()` (Epic B), `getConfigurationSummary()` (Epic C), and `checkMigrationsApplied()`/`getDatabaseVersion()` (`db-utils.ts`) are three distinct, non-overlapping functions, not competing implementations of the same check.
- **No duplicate branding source** — the primary color's code-only boundary (`src/app/globals.css`) was reaffirmed, not duplicated, by Sprint C1's Configuration Module (read-only status row, not a second editable copy).

## 6. What This Checklist Does Not Cover

- A literal second `git clone` into a scratch directory with a fully separate, freshly-provisioned Postgres instance — this environment has no local Postgres/Docker available to provision one. This sprint's dry run instead used a live, temporary state reset (`FrameworkConfig.setupCompleted` flipped to `false` and restored) against the existing database, which exercises the same code paths (the Setup Wizard's redirect guard, Finalize Setup, the readiness checks) but is not byte-for-byte identical to a genuinely new database. **MANUAL VERIFICATION REQUIRED**: the next real client's actual first deployment (a genuinely fresh Neon instance) should be spot-checked against this checklist once one exists, since a literal fresh-database run cannot be substituted for in this environment.
- Import Engine onboarding (Academic Structure / Student / Teacher import) — Epic D, not yet built, explicitly out of this sprint's scope.
- Everything [GO_LIVE_CHECKLIST.md](./GO_LIVE_CHECKLIST.md) (product) already covers: real infrastructure provisioning, imported data accuracy, staff training, go-live sign-off.
