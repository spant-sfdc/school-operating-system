# Client Customization Guide

**Purpose:** The literal checklist a new client repository needs completed before it's ready for deployment — the concrete deliverable of [Epic C — Client Customization Framework](./EPIC_ROADMAP.md#epic-c--client-customization-framework). Where [CONFIGURATION_GUIDE.md](../CONFIGURATION_GUIDE.md) answers "where does a given fact belong" in the abstract, this document answers "what does someone actually do, in order, to turn the master framework into School #2's repository." **This is a planning document naming the checklist — the checklist itself should be exercised as part of Epic C's own implementation, not assumed complete by writing it down here.**

---

## 1. Starting Point: What [PRODUCT_ARCHITECTURE.md § 15](../PRODUCT_ARCHITECTURE.md#15-architecture-review--does-pant-public-school-still-appear-inside-the-architecture)'s Existing Audit Already Found

That audit (conducted before Sprints 0–5's backend work existed) found exactly four points of school-specific coupling in the codebase at the time: `src/config/school.ts`, `package.json`'s `name` field, each page composite's `content.ts`, and `PROJECT_CONTEXT.md § 1`'s own opening framing. Epic C's first job is **re-running this audit**, not re-deriving it from scratch, because the backend built since then (Sprints 0–5) introduces new candidate coupling points that audit never checked:

- `prisma/seed.ts` — the generic-but-still-Pant-Public-School-shaped class/subject/teacher seed lists (11 classes Nursery–8, 10 subjects, 3 example teachers). These are **development fixtures**, not client data — a new client repository should not ship with them seeded into its production database at all (see [CLIENT_IMPLEMENTATION_PLAYBOOK.md § 2.1](./CLIENT_IMPLEMENTATION_PLAYBOOK.md#21-stage-by-stage) step 5's explicit "reference-data only" instruction).
- The literal string `"seed-school"` used as `School.schoolId` across every repository/service call in the seed script and any smoke tests — a hardcoded ID, not a config-sourced value.
- `docs/` itself — every document in this set narrates Pant Public School's own history (`IMPLEMENTATION_LOG.md`, `CHANGELOG.md`, `DECISIONS.md`). This is **correct and expected** to stay client #1-specific — a new client repository does not need Pant Public School's implementation history, but it does need `docs/engineering/ENGINEERING_PRINCIPLES.md` and the architectural/technical documents (`ARCHITECTURE.md`, `DEVELOPMENT_CONVENTIONS.md`, the `domain/`/`database/` sets) — see § 4 for the exact split.

## 3. What Should Never Need to Change

The entire point of the config/content/code boundary already established in [CONFIGURATION_GUIDE.md](../CONFIGURATION_GUIDE.md) — restated here because it's the thing that makes this checklist short:

- Every Marketing Section Library component (`PageHero`, `FeatureGrid`, etc.) — takes props, renders identically for any school's data.
- Every repository, service, DTO, and validation schema (`src/repositories/`, `src/services/`, `src/lib/validations/`) — universal business logic, per [ENGINEERING_PRINCIPLES.md](../engineering/ENGINEERING_PRINCIPLES.md); a second school's attendance percentage is calculated by the exact same code as the first's.
- The Prisma schema and every migration — the data model doesn't change per client; only the data inside it does.
- `docs/engineering/`, `docs/domain/`, `docs/database/`, `ARCHITECTURE.md`, `DEVELOPMENT_CONVENTIONS.md`, `AI_RULES.md`, `MASTER_PROMPT.md` — the framework's own operating rules, equally true for every client repository.

If a checklist item ever needs adding here that touches genuine application logic (not a config value or a content string), that is a signal the framework itself has a real single-tenant assumption baked in somewhere — worth flagging as a framework bug to fix in the master repository (benefiting every future client at once), not working around in one client repo alone.

## 4. Documentation Set — What Ships With a Client Repo, What Doesn't

| Document set                                                                                   |          Ships with every client repo?           | Reason                                                                                                                                                                                                                                                                         |
| ---------------------------------------------------------------------------------------------- | :----------------------------------------------: | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `docs/AI_RULES.md`, `MASTER_PROMPT.md`, `DEVELOPMENT_CONVENTIONS.md`, `ARCHITECTURE.md`        |                       Yes                        | The framework's own operating rules — identical for every client                                                                                                                                                                                                               |
| `docs/domain/`, `docs/database/`, `docs/engineering/`                                          |                       Yes                        | The data model and its physical/architectural reasoning — universal, not school-specific                                                                                                                                                                                       |
| `docs/product/` (this set)                                                                     |                Yes, as reference                 | A client repo doesn't run its own delivery — but keeping the playbook alongside the code that implements it avoids drift between "what the docs say" and "what a fresh clone actually needs"                                                                                   |
| `docs/PROJECT_CONTEXT.md`, `DECISIONS.md`, `IMPLEMENTATION_LOG.md`, `CHANGELOG.md`, `TASKS.md` |            **No — reset per client**             | These narrate Pant Public School's own build history. A new client repo should start its own, fresh versions of these — its own current-state, its own decision log, its own task board — not inherit this repository's history as if it were the new client's own.            |
| `docs/onboarding/` (Content Readiness Framework)                                               | Yes, as a **reusable framework**, reset to empty | Already explicitly designed to be reusable for any future school, per [PROJECT_CONTEXT.md § 5](../PROJECT_CONTEXT.md#5-completed-features)'s own description — a new client repo runs the same framework against its own pages, starting from its own zero-published baseline. |

This distinction — reused-as-is vs. reset-to-empty-but-same-framework vs. never-carried-over — is itself worth naming as a concrete Epic C deliverable: a documented "which docs get reset, which don't" table, so a future engineer cloning this repo doesn't have to guess.
