## D-001 — Three Roles Only for Version 1

- **Description:** Version 1 supports exactly three roles: Guest, Admin, Teacher.
- **Reason:** Traditional school ERPs support many roles (Admin, Teacher, Parent, Student, Accountant, Librarian). Supporting all of these from launch adds significant auth, permission, and UI complexity. Restricting to three keeps v1 scoped and shippable; Parent/Student portals can be layered on later without a rewrite given the role-segmented architecture.
- **Alternatives Considered:** Include a read-only Parent role from launch — rejected as unnecessary launch-critical complexity; deferred to post-launch roadmap.
- **Approved By:** Project Owner
- **Date:** 2026-07-17
- **Status:** ✅ Approved
- **Affected Documents:** [PROJECT_CONTEXT.md](./PROJECT_CONTEXT.md), [PRODUCT_REQUIREMENTS.md](./PRODUCT_REQUIREMENTS.md), [AI_RULES.md](./AI_RULES.md)

## D-003 — Design Direction: Stripe/Notion-Class Polish, Not Traditional ERP UI

- **Description:** UI/UX is modeled on Stripe Dashboard, Linear, Vercel Dashboard, and Notion — minimal, calm, accessible, usable without training.
- **Reason:** Existing school software (Fedena, PowerSchool, ERPNext Education) is functional but visually dated and often requires training. Daily users are teachers and school staff, not IT specialists; adoption depends on the product being effortless.
- **Alternatives Considered:** Follow conventional school-ERP visual patterns (dense tables, heavy sidebars, form-first workflows) for faster familiarity to admins who've used such systems before — rejected; optimizing for effortless daily use outweighs superficial familiarity.
- **Approved By:** Project Owner
- **Date:** 2026-07-17
- **Status:** ✅ Approved
- **Affected Documents:** [PROJECT_CONTEXT.md](./PROJECT_CONTEXT.md), [UI_DESIGN_SYSTEM.md](./UI_DESIGN_SYSTEM.md)
- **Future Review Required:** No — foundational design principle; revisit only if user feedback strongly contradicts it post-launch.

## D-004 — Tech Stack Selection

- **Description:** Next.js 15 + React 19 + TypeScript + Tailwind CSS v4 + Shadcn UI + Framer Motion + Lucide Icons on the frontend; Next.js API routes + PostgreSQL + Prisma on the backend; Auth.js for authentication; Vercel + Cloudinary + GitHub for hosting/media/source control.

Remix instead of Next.js — rejected, Next.js has stronger Vercel-native deployment story and larger Shadcn ecosystem alignment.

## D-007 — `src/` Directory Layout

- **Description:** Application code (`app/`, `components/`, `lib/`, `types/`, `hooks/`) is nested under a `src/` directory. `prisma/`, `public/`, `docs/`, and root config files remain at the repository root. Internal composition of `app/`, `components/`, and `lib/` follows [ARCHITECTURE.md § Folder Structure](./ARCHITECTURE.md#2-folder-structure) unchanged — only the top-level container changed.
- **Reason:** Phase 0B.1 scaffolding instructions explicitly requested a `src/`-based, feature-first layout. This is a standard, Next.js-supported convention that separates application code from repository-root configuration/tooling files, which is valuable now that ESLint, Prettier, Husky, VSCode, and editor config files live at the root alongside `docs/`. `ARCHITECTURE.md`'s originally documented tree did not include a `src/` wrapper; this decision formally supersedes that detail while leaving every other structural decision (route groups, component hierarchy, lib organization) intact.
- **Alternatives Considered:** Keep `app/`, `components/`, `lib/`, `types/` at the repository root exactly as originally drafted in `ARCHITECTURE.md` — rejected because it directly conflicts with the explicit Phase 0B.1 scaffolding instruction, and mixing application source with the growing set of root-level config files reduces clarity as the project grows.

## D-009 — Phase 0B.1 Finalization: Dependency Cleanup and Runtime Pinning

- **Description:** Removed `@base-ui/react` and `class-variance-authority` (dependencies of the `shadcn init`-generated `Button` component, which was itself removed under D-008's "No Components" reading — these two were left behind, orphaned, with zero imports anywhere in `src/`). Reclassified `shadcn` from `dependencies` to `devDependencies` (it is a build-time CLI, never imported by application code). Pinned the project runtime via `.nvmrc` (`22`) and `package.json engines` (`node: >=22 <23`, `pnpm: >=10`). Upgraded the global `pnpm` installation from 9.15.9 to 10.34.5 (compatible with Node 22's `node:sqlite`, which pnpm 11 requires and which is available under Node 22.23.1) so the pinned engines constraint is actually satisfiable, not just aspirational. Added `pnpm-workspace.yaml` with `allowBuilds: { prisma: true }`, pnpm 10's explicit opt-in for dependency postinstall scripts.

Leaving `@base-ui/react`/`class-variance-authority` installed with no importer anywhere is dead weight with no future-phase justification (unlike, say, `next-intl`, which is unwired but explicitly approved tech stack). Pinning the runtime converts the informal Node-22 workaround from the previous session into an enforceable, documented constraint instead of tribal knowledge.
- **Alternatives Considered:** Leave the engines field aspirational and keep working around it locally — rejected; an unenforced pin is not a pin, and the whole point of this finalization pass is to leave the repository in a state that doesn't depend on undocumented local workarounds. Set `engine-strict` off / omit the `pnpm` engine constraint to avoid the local mismatch entirely — rejected; the task explicitly specified both fields, and the correct fix for an environment mismatch is to align the environment, not loosen the constraint. Remove `shadcn` entirely instead of reclassifying — considered, but keeping it pinned as a devDependency avoids version drift for `pnpm exec shadcn add` calls in Phase 1.

## D-012 — Marketing Section Library Location: `components/website/sections/`, Not `src/features/`

- **Description:** Phase 1B's instructions asked for a mandatory architecture review of whether `src/components/website/` should remain or move to a suggested `src/features/website/marketing/sections/` feature-first structure. Outcome: **kept `src/components/website/` in place** (zero existing files moved, renamed, or behaviorally changed) and added a new `sections/` subfolder inside it for the 15 new reusable marketing primitives (`PageHero`, `SectionHeader`, `ContentContainer`, `SectionDivider`, `ImageText`, `FeatureGrid`, `StatisticsGrid`, `Timeline`, `QuoteBlock`, `CTASection`, `FAQAccordion`, `BadgeGroup`, `Callout`, `ResponsiveImage`, `AnimatedSection`), each in its own folder (`Component.tsx`, `Component.types.ts`, optionally `Component.constants.ts`, `README.md`, `index.ts`) — adopting the task's suggested per-component structure without adopting its suggested top-level `features/` tree.
- **Reason:** A full `components/` → `features/` migration would introduce a second, parallel organizational philosophy alongside the already-approved, freshly-reviewed role-segmented model (`components/{ui,shared,website,admin,teacher}` — [ARCHITECTURE.md § Component Hierarchy](./ARCHITECTURE.md#6-component-hierarchy), [D-011](#d-011--componentswebsite-folder)). Admin and Teacher will eventually need their own reusable composite libraries too (dashboards, data grids); if only `website/` moved to `features/`, future contributors would face two incompatible mental models for the same kind of problem, which is a maintainability _regression_, not the improvement the task explicitly asked for. The task's own framing — "the approved feature-first architecture" — does not match any actually-approved document; no decision or `ARCHITECTURE.md` revision ever established a `features/` tree, so proceeding as if it were approved would have been building on a false premise rather than flagging it. The task's phrasing "Refactor **ONLY IF REQUIRED**" and "You are **NOT** designing new architecture" both point away from a full migration. The genuinely valuable part of the suggestion — self-contained component folders with co-located types/constants/README/index, which directly supports the task's own "future Storybook support" goal — was adopted in full for the _new_ components, at no risk to existing, working, just-approved code.
- **Alternatives Considered:** Migrate all of `components/website/` (including `SiteHeader`, `SiteFooter`, `Hero`, `MobileNav`, `LanguageSwitch`, `AnimatedBackground`, `StatStrip`) into `src/features/website/marketing/sections/` and `shared/` as the task's architecture diagram suggested — rejected for the reasons above; also would have violated "this refactor should be invisible to the end user" far more riskily than necessary (moving 7 working, tested files for a purely organizational preference, with real regression risk, versus moving zero files). Leave new marketing primitives as flat files directly in `components/website/` with no `sections/` subfolder — rejected; 15 additional flat files would make `components/website/` (already 8 files) hard to scan, and would forgo the legitimate, low-risk per-component-folder benefit (co-located types/README, Storybook-readiness).
- **Approved By:** Project Owner (via Phase 1B implementation — architecture review was mandatory per task instructions; this record satisfies that requirement)
- **Date:** 2026-07-17
- **Status:** ✅ Approved

## D-013 — Phase 1B.1 Architecture Review: Reaffirmed Role-Segmented `components/`, Added a Feature-Subgrouping Threshold Rule

- **Description:** A dedicated Staff-Architect-level review re-examined `components/website/` vs. a `src/features/website/` migration from first principles (not by deferring to D-012), scored across 20 dimensions (maintainability, scalability, DX, testing, Storybook, CMS integration, feature isolation, code reuse, performance/tree-shaking, refactoring cost, discoverability, team collaboration, and six forward-looking product scenarios: future SaaS, multi-school, API integration, mobile app, parent portal, teacher/admin portal). **Outcome: reaffirmed D-012 — role-segmented `components/{ui,shared,website,admin,teacher}` remains correct, no migration performed.** New addition beyond D-012: an explicit, testable graduation rule — **a role folder should adopt `sections/`-style feature-subgrouping (per-component folders, same convention as `website/sections/`) once it exceeds roughly 8 components spanning more than 2 distinct domains** (e.g., `components/admin/attendance/`, `components/admin/marks/` once `components/admin/` reaches that size) — rather than an unbounded "revisit later."
- **Reason:** Of the 20 dimensions scored, 18 were neutral-to-favorable for the existing role-segmented structure or genuinely orthogonal to the folder-axis question entirely (multi-tenancy, future mobile app, future API integration, and future SaaS are data-layer/routing/infra concerns — already correctly flagged as open questions in [ARCHITECTURE.md § 9](./ARCHITECTURE.md#9-open-architectural-questions) — not folder-naming concerns; a future React Native mobile app would not consume this Next.js `components/` tree under either scheme unless a much bigger, separately-decided monorepo/shared-UI-kit strategy is adopted). This product's primary, product-level axis of variation is **role** (Guest/Admin/Teacher, hard-fixed by [D-001](#d-001--three-roles-only-for-version-1)), not an open-ended feature list — role-first structure encodes a stable, guarded product decision; feature-first would encode feature boundaries for Admin/Teacher that don't exist yet (both folders are still empty), which is speculative design for code that hasn't been written. The one dimension where feature-first has a genuine, uncontestable edge is **refactoring cost** — today, with 15 components and zero page consumers, is the cheapest this migration will ever be. That is real and is recorded here rather than hidden, but it is not, on its own, sufficient reason to reverse a decision made one phase earlier with no new evidence besides the same suggestion restated more forcefully — that would be thrashing, not engineering judgment. The threshold rule closes the actual gap in D-012: D-012 explained why not to migrate `website/` alone, but never stated when subgrouping becomes warranted for a role folder in general.
- **Alternatives Considered:** Migrate now, while it's cheapest — rejected; the boundaries a migration would encode (which components belong to which "feature") aren't knowable yet for Admin/Teacher, since neither has been built. Migrating speculatively risks guessing wrong and paying the refactor cost twice. Defer the question indefinitely with no trigger condition (D-012's original position) — rejected as incomplete; "revisit later" without a concrete threshold is exactly the kind of vague deferral that produces silent architecture drift, the same failure mode [IMPLEMENTATION_LOG.md § Phase 0A.1](./IMPLEMENTATION_LOG.md#2026-07-17--phase-0a1-documentation-consolidation--ai-operating-system) already identified once in this project's history.
- **Approved By:** Project Owner (via Phase 1B.1 architecture review instructions — "produce an engineering decision")
- **Date:** 2026-07-17
- **Status:** ✅ Approved
- **Affected Documents:** [ARCHITECTURE.md](./ARCHITECTURE.md), [IMPLEMENTATION_LOG.md](./IMPLEMENTATION_LOG.md)
- **Future Review Required:** Yes — trigger condition is explicit: revisit the moment any single role folder (`admin/`, `teacher/`, or `website/` outside `sections/`) exceeds ~8 components across 2+ distinct domains.

## D-014 — SectionDivider YAGNI Simplification

- **Description:** Removed the `minimal` and `decorative` variants from `SectionDivider`, reducing its `variant` prop from 4 options (`light`/`dark`/`minimal`/`decorative`) to 2 (`light`/`dark`).
- **Reason:** Neither variant had a spec, a consumer, or a design-system reference — both were invented during Phase 1B build-out, not requested. Inspection during the Phase 1B.1 component-by-component YAGNI review found both rendered the exact same `border-border` color as `light`; their only actual difference was DOM shape (short centered line / line-dot-line), meaning the `variant` prop was silently conflating two unrelated concerns (tone: light vs. dark background; shape: full rule vs. decorative accent) under one name. This is genuine unjustified surface area — untested-by-real-use API shape that a future consumer would have had to guess the intent of. `light`/`dark` are kept because they map to a real, present need (a divider usable on both `background` and `surface-muted` sections).
- **Alternatives Considered:** Keep all 4 variants "in case a future page wants them" — rejected; per [AI_RULES.md § Quality Bar](./AI_RULES.md#5-quality-bar) ("no unnecessary abstraction... do not build generic frameworks for one use case") and [PROJECT_GUARDRAILS.md § G-3](./PROJECT_GUARDRAILS.md#1-the-core-guardrails), speculative API surface with zero consumers is exactly what YAGNI review exists to catch — re-adding a variant later (a `<hr>` with different Tailwind classes) costs minutes, not a redesign.

## D-015 — Permanent `/dev/playground` Route Replaces Temporary Preview-and-Delete Pattern

- **Description:** Added `src/app/dev/playground/page.tsx` — a permanent route composing every component in the Marketing Section Library with representative sample data, guarded by `if (process.env.NODE_ENV === "production") notFound();` at the top of the page. It builds and typechecks as part of the normal app, but statically resolves to a genuine 404 in a production build (verified: `pnpm build && pnpm start` returns HTTP 404 for `/dev/playground`; the route works normally under `pnpm dev`). This replaces the Phase 1B pattern of building a temporary preview route by hand each session and deleting it before finishing.
- **Reason:** Phase 1B's manual-verification step required building `src/app/dev-preview-temp/` from scratch, including re-discovering that Next.js excludes underscore-prefixed folders from routing (a mistake that cost real time — first attempt at `_dev-preview` silently 404'd). A permanent, correctly-named route pays that setup cost exactly once instead of once per phase, and gives every future phase (targeting 100+ components) a standing place to visually verify new components without inventing a new throwaway page each time. The alternative — adopting Storybook, which `ARCHITECTURE.md § 6` already flags as a future consideration — is a larger, separate decision (new dependency, new dev-only tooling surface, its own config and addon ecosystem) that deserves its own justification and its own phase, not a default reached for inside an architecture review; a plain guarded Next.js route needed zero new dependencies and solves the immediate, recurring pain.
- **Alternatives Considered:** Keep the build-and-delete temporary preview pattern — rejected; it repeats avoidable setup cost and avoidable mistakes (the underscore-folder 404) every single phase, for a library explicitly meant to grow past 100 components. Adopt Storybook now — rejected for this session; heavier tooling change than the review's scope justifies, and the codebase has no consumer pain yet that a lightweight route doesn't already solve (revisit if/when component count or team size makes Storybook's controls/addons genuinely worth the setup cost). Guard the route via `middleware.ts` instead of an in-page check — considered and rejected as unnecessary indirection; no `middleware.ts` exists in the codebase yet, and introducing one solely for this would be a bigger architectural footprint than a single `notFound()` call for a single dev-only page.

## D-025 — Physical Database Review: Time-Ordered IDs for High-Volume Tables, Partial Unique Indexes, Partitioned Audit Log

- **Description:** Published the physical database review — `docs/database/{README,DATABASE_REVIEW,INDEXING_STRATEGY,CONSTRAINT_STRATEGY,ENUM_STRATEGY,AUDIT_STRATEGY,SOFT_DELETE_STRATEGY,QUERY_PATTERNS,MIGRATION_PLAN,PERFORMANCE_GUIDELINES,PRISMA_IMPLEMENTATION_GUIDE}.md` — reviewing [docs/domain/DATABASE_SCHEMA.md](./domain/DATABASE_SCHEMA.md)'s illustrative shape as a physical Postgres design, ahead of Epic B's actual Prisma implementation. Four physical decisions carry the most weight: (1) `AttendanceRecord`, `MarksRecord`, and `AuditLog` — the three genuinely high-volume, append-heavy tables — should use time-ordered IDs (ULID/UUIDv7) instead of the domain model's uniform `cuid()`, for B-tree insert locality at the row counts these tables will reach; every other table stays `cuid2`. (2) Two business-rule invariants the domain model correctly identified but explicitly left as "application-enforced" — `AcademicYear.isCurrent` and `TeacherAssignment.isClassTeacher` each being unique per scope — should instead be schema-enforced via Postgres partial unique indexes. Three physical gaps were also found in the domain model's own illustrative schema: `Section` and `Subject` are each missing a uniqueness constraint, and `TransferCertificate.tcNumber` was drafted globally unique instead of per-school — the last of which would actively break at Epic H's first second-tenant onboarding, not just a single-tenant data-integrity gap.
- **Reason:** Each of the four decisions follows the same "cheap now, expensive later" argument this project has applied consistently since [D-018](#d-018--centralized-configuration-layer-srcconfig)'s config layer and [D-022](#d-022--school-domain-model-board-agnostic-enrollment-centric-rte-aware)'s `Enrollment`-as-hub design: retrofitting a primary-key strategy, a missing constraint, or table partitioning after real multi-year attendance/marks data exists is materially more expensive and riskier than deciding it correctly before the first migration, when the cost is a few lines in a schema file. Silently correct the three schema gaps found (`Section`/`Subject`/`TransferCertificate`) inside this review's own documents — rejected; corrections to [docs/domain/DATABASE_SCHEMA.md](./domain/DATABASE_SCHEMA.md) belong in that document's own future edit, not smuggled into a review that's explicitly reviewing, not redesigning, the domain model.

## D-036 — Sprint B2 Administration & User Management Foundation: `User.mustChangePassword` Column, Centralized `src/lib/authorization/`, Orchestrated (Not Modified) `registerTeacher()`/`createIdentityUser()`, Cookie-Flashed Temporary Passwords

## D-038 — Sprint B2.2 Post-Login Redirect: Single `resolvePostLoginRedirect()` Resolver Replaces Three Independent Role-Mapping Implementations; `auth()` Does Not See a Same-Request `signIn({redirect:false})` Cookie in This Next.js Version

- **Description:** After D-037's credential fix, a real login correctly established a session but then landed on `/` — the public homepage, which has no route guard — instead of `/change-password` (when forced) or a role-appropriate landing page. Root cause: `src/app/(auth)/login/page.tsx`'s `loginAction` computed `redirectTo` as `callbackUrl || "/"`, and Sprint B1 never had a role-based landing page to redirect to instead (`ROUTES.md § 2`'s own documented gap). Since `/` is public, nothing downstream (middleware, layouts) ever re-routed an authenticated visitor who landed there.

  **Fix, in two parts:**

  **(1) One authoritative resolver.** Added `resolvePostLoginRedirect(subject, requestedPath?)` in `src/lib/authorization/redirect.ts` — given `{ accessLevel, mustChangePassword }`, returns `/change-password` unconditionally if `mustChangePassword` is true (overriding any requested deep link), otherwise the requested path if one was given (preserving Sprint B1's "return to what you were trying to reach" `callbackUrl` behavior for the non-forced case), otherwise a role-based landing page (`/admin` for `ADMIN`, `/teacher` for `TEACHER`). Before this fix, the "which role goes where" mapping was about to exist in **three independent places**: the login action (new), `change-password/actions.ts`'s post-change redirect (already inline `session.accessLevel === "ADMIN" ? "/admin" : "/teacher"`), and implicitly assumed by the Admin/Teacher layouts' own `mustChangePassword` check. All three now call the same function (the layouts import only its `CHANGE_PASSWORD_PATH` constant, since they only need that one fixed destination, not the full role-mapping logic) — the destination can now only ever be wrong in one place, not silently drift across three.

  **(2) A same-request `auth()`-after-`signIn()` re-read does not work in this environment — discovered by testing, not assumed.** The first implementation called `signIn("credentials", {..., redirect: false})`, then `await auth()` immediately after in the same Server Action to read back the freshly-established session and compute the redirect. `@auth/core`'s own source confirms `signIn()` sets the session cookie via `cookies().set()` unconditionally, before checking the `redirect` option — so this looked sound. **Live testing showed otherwise**: the resulting `auth()` call returned no session (`session?.user?.accessLevel` was `undefined`) even though the `Set-Cookie` header for a valid `authjs.session-token` was present in the actual HTTP response, confirmed by inspecting the raw response and cookie jar directly. **Fixed** by not depending on this timing at all: `loginAction` now calls `authenticateUser()` (the same function the Credentials provider's own `authorize()` calls internally, `src/services/auth/auth.service.ts`) before `signIn()`, computes the destination from that DTO via `resolvePostLoginRedirect()`, and passes it to `signIn()` as `redirectTo` with the default `redirect: true` behavior (which throws internally to perform the actual navigation, exactly as Sprint B1's original code already relied on). This means `authenticateUser()` runs twice per login (once here, once inside `authorize()`) — an accepted, explicit tradeoff, not an oversight; see Alternatives Considered.

  **Verified end to end against a live, freshly built production server**, via `curl` mirroring the real login form's exact field/action-id shape: bootstrap login (`mustChangePassword: true`) → `/change-password` directly, no stop at `/` → submitting a new password → `/admin` → `GET /admin`/`GET /admin/users` both 200 → sign-out → session cleared, `/admin` → 307 → logging in again with the new password → directly `/admin`, no `/change-password` detour. Repeated the same sequence for a Teacher account (`sunita.verma@example.com`) — forced change lands on `/teacher`, not `/admin`, confirming the role branch (not just the `mustChangePassword` branch) is exercised correctly. Also regression-tested: a wrong password still shows the existing "Incorrect Login ID or password" error, and an unauthenticated deep link to `/admin/users` still round-trips through `callbackUrl` back to `/admin/users` after a non-forced login — neither pre-existing behavior was broken by this change.

- **Reason:** "Design the cleanest architecture, don't implement this in multiple places" was the task's own explicit instruction — a single named resolver function, imported everywhere a post-login or post-password-change destination is needed, is the direct answer. The `redirect: false` approach was abandoned only after live evidence contradicted the source-reading-based assumption that a same-request cookie mutation would be visible to an immediately-following read — matching this project's "verify by executing" discipline (D-031, D-032, D-035, D-037), not reverted on a hunch.
- **Alternatives Considered:** Keep `signIn({redirect:false})` and work around the missing session by decoding the JWT cookie directly out of the response instead of calling `auth()` — rejected; would mean hand-rolling session-token parsing/verification this codebase already has a correct, tested implementation of (`auth()`/`resolveActiveSessionUser()`), a duplicate, lower-level reimplementation for no real benefit over calling `authenticateUser()` a second time. Put the redirect decision inside Auth.js's own `callbacks.redirect` hook in `src/lib/auth/config.ts` — rejected; that hook receives only `{ url, baseUrl }`, not the authenticated user's role/`mustChangePassword`, and coercing business logic into a generic framework hook not designed to carry it would be harder to follow than a small, explicitly-named, directly-testable function. Have the Admin/Teacher layouts also call `resolvePostLoginRedirect()` for their `mustChangePassword` check (fully collapsing all three sites into identical calls) — rejected; a layout already knows it wants exactly `/change-password`, unconditionally, when the flag is true — it doesn't need the resolver's role-branching logic, only the shared path constant, so importing just `CHANGE_PASSWORD_PATH` is the more honest reflection of what each site actually needs. Accept the double `authenticateUser()` call's extra Argon2id verify as a performance concern and cache the first result across the two calls — rejected as premature; login is not a hot path, and the two calls are already independent, correctly-scoped operations (a pre-flight decision vs. the provider's own authoritative check) — introducing a cache to avoid one extra hash comparison during sign-in would be optimizing a cost this project has no evidence is actually a problem.

## D-043 — Sprint D1 Import Engine Architecture & Foundation: Two New Tables (`ImportBatch`/`ImportRow`, Each Independently Justified), Chunked Commit Is One Transaction Per Row Not Per Chunk (Matching `TRANSACTION_BOUNDARIES.md § 4`'s Rollover Precedent), Entity-Agnostic Extension Points Only — No Real Importer Built

- **Description:** Business Workflow Review — "a school has given us 12 Excel files, how should this framework safely migrate them" — performed before any code, per this sprint's own instruction. Mapped the real answer: 12 files is not one mega-import — each file becomes its own `ImportBatch` (one entity type per batch), sequenced by the Admin in an order that respects real dependencies (Classes/Sections/Subjects before Students/Teachers, since a student needs a section to enroll into — `IMPORT_ENGINE_STRATEGY.md`'s own note), each walking the same Upload → Detect → Map → Validate → Preview → Commit → Audit → History → Report lifecycle regardless of which of the 12 files it is. This is the concrete case for "one engine, not one importer per entity" (`IMPORT_ENGINE_STRATEGY.md § 1`) — the lifecycle and its state machine are identical across all 12 files; only a small, declarative per-entity profile (validator + commit handler, see below) would ever differ, and this sprint builds none of those profiles.

  **Database Review — two tables, each independently justified:**

  - `ImportBatch` — one row per import run, the provenance record `IMPORT_ENGINE_STRATEGY.md § 2.6` requires to answer "show me everything that came from the March 2027 migration," a question `AuditLog` alone can't answer efficiently since it isn't grouped by import run.
  - `ImportRow` — genuinely required, not a convenience table: without per-row state, a chunked commit has no way to resume after an interruption except recomputing from the raw file every time, defeating resumability's purpose. Also carries row-level validation errors (a JSON array column, not a third `ImportError` table — nothing in this sprint's scope needs to query errors independently across batches, so a separate table would be schema weight with no present justification).

  Both follow `SOFT_DELETE_STRATEGY.md § 1`'s third category — "historical fact/measurement, no delete mechanism at all," the same category as `AttendanceSession`/`AttendanceRecord`/`AuditLog`. `ImportRow` uses a time-ordered UUIDv7 PK (`DATABASE_REVIEW.md § 11`'s precedent for high-insert-volume tables) since a single large import can genuinely produce thousands of rows; `ImportBatch` uses `cuid()` (low volume, one row per run).

  **Transaction Review, against `TRANSACTION_BOUNDARIES.md § 4`'s own Academic Year Rollover precedent — the same shape of problem as Import (a large batch operation), explicitly not the same shape as a 40-row attendance submission:** the naive design (one transaction per chunk of N rows) was considered and rejected — `TRANSACTION_BOUNDARIES.md § 4` names exactly this failure mode as the bad outcome to avoid ("a single failure partway through would roll back potentially hundreds of already-correct decisions"). `commitImportBatchChunk()` instead commits **one row per transaction**, inside a chunk-sized loop — a bad row's failure marks only that row `FAILED` and never touches the other rows in the same chunk. "Chunking" here means "process N rows before returning progress" (a batching of _work_, useful for a future queue-worker or client-polling caller), not a batching of _transactions_. Resumability follows directly: `findNextChunkToCommit()`'s own `WHERE status = 'VALID'` naturally excludes already-`COMMITTED` rows, so a second call after an interruption picks up exactly where the first left off — verified live, not assumed (see Verification Results).

  **Import Service — entity-agnostic orchestration, zero embedded entity knowledge:** `createImportBatch()` (Upload), `ingestImportRows()` (Detect+Map — one bulk insert, not N), `validateImportBatch()` (Validate — Business Validation only, via a caller-supplied `ImportRowValidator`), `previewImportBatch()`/`skipInvalidRows()` (Preview — the mandatory gate `IMPORT_ENGINE_STRATEGY.md § 2.4` requires; no direct upload-and-commit path exists), `commitImportBatchChunk()` (Commit — resumable, per-row-transactional, driven by a caller-supplied `ImportRowCommitHandler`), `getImportReport()`/`listImportBatches()`/`getImportBatchDetail()` (Report/History).   **Validation Strategy, four layers named and separated, per this sprint's own instruction:** File validation (parseability/size — out of this sprint's scope, no parsing library added, see Alternatives Considered) → Column validation (does every required target field have a mapped source column — a future importer's own concern, since "required" is entity-specific) → Business Validation (the `ImportRowValidator` contract this sprint builds) → Database validation (does a mapped value resolve to a real row, e.g. a student's `className` to an actual `SchoolClass` — a future importer's own concern, requires entity-specific repository knowledge this foundation deliberately doesn't have).

  **AI Preparation, extension points only, nothing implemented:** `suggestColumnMappings()` — a naive, deterministic, case-insensitive exact-match column-detection helper (the Detect stage's real, if minimal, implementation) with a `confidence: 0 | 1` field whose range is reserved for a future AI-backed implementation to return genuine partial-confidence scores as a drop-in replacement. "Data quality analysis" and "government format templates" are named, not designed or coded — building either now would be exactly the "AI mapping"/entity-specific scope this sprint's own instructions forbid.

  **Security Review:** Upload limits/malicious files — named as the future Upload endpoint's own concern (no endpoint exists yet); duplicate imports — `ImportBatch.sourceFileHash` (indexed, not unique) plus `findPriorBatchForFile()` lets a future Upload UI warn, not block, on a repeat file; idempotency — `createImportRows()`'s bulk insert and `commitImportBatchChunk()`'s resumable, already-committed-rows-excluded design are both naturally idempotent against a repeated call; partial failures — the per-row-transaction design above is the direct answer; permissions — `canManageImports()` (Admin-only, same reasoning as every other administrative capability in `src/lib/authorization/permissions.ts`).

  **`/admin/imports`** — a deliberately minimal Server Component History list (mirrors `/admin/audit`'s established list-page pattern), reusing `listImportBatches()`. Shows "No imports yet" today, since no entity importer exists to ever create a real batch — proves the repository/service/route layer works end to end without building any Upload/Map/Validate/Preview/Commit UI (all of which would require an actual file-parsing dependency and at least one real entity profile, both explicitly out of this sprint's scope).

  **Verified end to end via a live scratch script** (not code review alone): created a real `ImportBatch`, ingested 4 rows, validated with a synthetic validator (correctly flagged 1 invalid row), previewed (correct row counts and grouped error summary), skipped the invalid row, committed in two chunks of 2 (chunk 1: 2 rows committed; chunk 2: resumed correctly, processed only the 1 remaining row, one row deliberately made to fail its commit — confirmed it failed in isolation without affecting the other 2 already-committed rows), confirmed the batch settled to `PARTIALLY_COMPLETED` with correct counts, confirmed the History list/detail functions both surface the batch correctly, confirmed 5 `AuditLog` entries were written across the lifecycle (`CREATE` at Upload, `UPDATE` at each subsequent stage transition) reusing the existing `writeAuditLog()` mechanism with zero new audit infrastructure. All test rows (`ImportBatch`/`ImportRow`/`AuditLog`) were then deleted via a scratch-script-only raw Prisma call — not through any application feature, since neither table has (or should have) a delete mechanism — restoring the database to its pre-test state.

- **Reason:** Building the orchestration layer, schema, and extension-point contracts before any real entity importer exists lets every future importer (`StudentImporter`, `TeacherImporter`, `AttendanceImporter`, ...) be a small, declarative plugin against an already-proven state machine, rather than each one reinventing chunking/resumability/audit/history from scratch — directly the "one engine, not one importer per entity" principle `IMPORT_ENGINE_STRATEGY.md § 1` already established. Per-row transactions (not per-chunk) is the one place this sprint deviated from a naive reading of "chunked commit" toward the pattern `TRANSACTION_BOUNDARIES.md § 4` already prescribed for the structurally identical Rollover problem — verified live specifically because "does one bad row roll back its neighbors" is exactly the kind of behavior that looks correct in review and fails only when actually exercised.
- **Alternatives Considered:** A third `ImportError` table, separate from `ImportRow.validationErrors` — rejected; nothing in this sprint's scope needs to query errors independently across batches, and `docs/database/` `README.md`'s own "every table must justify its existence" discipline means not building it speculatively. One transaction per chunk (not per row) for Commit — rejected directly by `TRANSACTION_BOUNDARIES.md § 4`'s own named failure mode. ## D-044 — Sprint D2 Academic Structure Importer: First Real Importer Built as the Reference Architecture; Two Genuine Gaps Found in Sprint D1's Own Foundation and Fixed (`ImportRowValidator` Widened Async, `academic.service.ts` Gains Transaction Passthrough); `ImportEntityType` Simplified to One `ACADEMIC_STRUCTURE` Value

- **Description:** Business Workflow Review — "a school has sent us one Excel file containing Classes, Sections, Subjects, Academic Years, how should the system safely migrate it" — mapped the pipeline before any code: normalized rows (parsing stays outside this importer, per this sprint's own "do not tie the importer to CSV" instruction) → Data Profiling (raw-row-level duplicate/missing/unknown detection) → grouping (one row per source spreadsheet line collapses into one row per unique class, carrying its full section list, and one row per unique subject — matching what `createSchoolClassWithSections()`/`createAcademicSubject()` actually expect) → Business+Database Validation → Preview → Commit (chunked, per-row-transactional, reusing Sprint D1's `commitImportBatchChunk()` unchanged) → Audit → Report → History.

  **Two genuine gaps found in Sprint D1's own foundation, only surfaced by actually building the first real importer against it — exactly what a "reference architecture" sprint is for:**

  1. **`ImportRowValidator.validateRow()` was synchronous.** D1 specified it that way assuming Business Validation only needs the row's own data. This sprint's own explicit validation requirements ("Duplicate Classes," "Unknown Academic Year") are genuinely Database Validation — they require a query. Widened to `ImportRowValidationResult | Promise<ImportRowValidationResult>` (additive — any hypothetical D1-era synchronous validator still satisfies the new type), and `validateImportBatch()`'s calling loop now `await`s the result.
  2. **`createSchoolClassWithSections()`/`createAcademicSubject()` had no way to join an existing transaction.** Both open their own internal `db.$transaction()`, but `commitImportBatchChunk()` (D1) opens its own per-row transaction and passes it into `ImportRowCommitHandler.commitRow(rawData, actorUserId, tx)` specifically so the entity write and the `ImportRow` status update land atomically together. Calling either service as-is from inside `commitRow()` would silently open a **second, independent** transaction — the entity could be created successfully while the `ImportRow` status update fails separately, and on retry the commit handler would call `createSchoolClassWithSections()` again, which would throw "already exists" and mark a row that actually _succeeded_ as `FAILED`. This is a real idempotency bug, not a hypothetical — traced by reasoning through the retry path, not discovered by accident. **Fixed**: both functions gained an optional `tx?: Prisma.TransactionClient` parameter (the same passthrough pattern every repository in this codebase already uses) — when provided, the function writes directly to it instead of opening its own transaction; when omitted, behavior is byte-for-byte identical to before. `academic.service.ts` is Sprint 2 (Foundation Phase) code, never declared frozen the way Epic B was — this is additive, not a frozen-module modification.

  **`ImportEntityType` simplified**: D1 shipped `SCHOOL_CLASS`/`SECTION`/`SUBJECT` as three separate enum values; `EPIC_ROADMAP.md`'s own Epic D Implementation Order item 6 and `IMPORT_ENGINE_STRATEGY.md` both describe "Academic Structure" (Classes/Sections/Subjects together) as **one** import type — a school's academic-structure spreadsheet is one file, not three uploads. Confirmed via grep that nothing outside D1's own scaffolding referenced the three-way split (zero real usage to preserve), so replaced them with one `ACADEMIC_STRUCTURE` value rather than leaving three permanently-unused enum members as clutter — "architecture quality over speed," applied to willingness to correct a one-sprint-old design once it's proven wrong, not just to writing new code carefully.

  **Column Alias Dictionary** (`aliases.ts`) — a small, curated map (Class/Std/Standard → `className`, Section/Division → `sectionName`, Subject/Course → `subjectName`, deliberately not exhaustive, per this sprint's own "do not over-engineer" instruction) checked before falling back to D1's generic exact-match `suggestColumnMappings()`. **Import Template support** — architecture only, no UI, per this sprint's instruction: D1 already built the storage (`ImportBatch.columnMapping`) and lookup (`findMostRecentBatchByType()`, unused until now); `getAcademicStructureColumnMappingTemplate()` is the first real caller, returning a prior batch's own mapping if one exists, else an identity default.

  **Data Profiling** (`src/services/import/profiling.ts`) — built as a genuinely reusable, entity-agnostic, pure function (`buildDataProfile()`) in the Import Engine foundation itself, not buried inside the Academic Structure importer: duplicate detection via a caller-supplied key-field list, missing-value detection via a caller-supplied required-field list, "unknown value" detection via a caller-supplied set of known-good values the _caller_ resolves from the database first (keeping the profiler itself free of I/O, trivially reusable by a future Student/Teacher/Attendance importer without depending on Academic Structure's own module).

  **Verified end to end via a live scratch script, with real (not synthetic) entity creation** — unlike D1's own verification, which used synthetic validator/handler functions since no real entity existed yet to create: 7 raw rows (including a deliberately duplicate section, a duplicate subject, an unknown academic year, and a class — `Nursery` — already seeded by `prisma/seed.ts`) → grouped into 4 rows → Data Profiling correctly flagged 2 duplicate raw rows and 1 unknown-year reference → Validation correctly marked 2 rows `INVALID` (unknown year; already-exists class) and 2 `VALID` → Preview showed correct counts and grouped errors → Skip → Commit produced exactly 2 committed, 0 failed, batch `COMPLETED` → direct database queries confirmed a real `SchoolClass` with exactly 2 (deduped) `Section` rows and a real `Subject` were created **through the actual, unmodified `createSchoolClassWithSections()`/`createAcademicSubject()` services** (not bypassed) → 4 entity-level `AuditLog` entries (`SchoolClass:CREATE`, 2×`Section:CREATE`, `Subject:CREATE`) confirmed those services' own audit logging fired normally → 5 `ImportBatch`-level `AuditLog` entries confirmed the batch's own lifecycle audit trail → the batch appeared correctly in `listImportBatches()` (the `/admin/imports` History page's own data layer, unchanged from D1 — "no special handling," exactly as instructed). All test data (`SchoolClass`, `Section`s, `Subject`, `ImportBatch`, `ImportRow`s, `AuditLog` entries) removed afterward via a scratch-script-only raw cleanup, restoring the database to its pre-test state.

- **Reason:** A "reference architecture" sprint's job is specifically to stress-test the prior sprint's foundation against real requirements and fix what doesn't hold up — finding and fixing the async-validator and transaction-passthrough gaps _now_, while only one importer exists, is far cheaper than discovering them independently in three or four future importer sprints each working around the same two problems differently.
- **Alternatives Considered:** Keep `ImportRowValidator.validateRow()` synchronous and pre-fetch all needed reference data (existing classes, existing subjects, valid academic years) into an in-memory lookup before validation begins — rejected; this shifts complexity into every future importer's own caller code instead of the shared contract, and for a school's realistic academic-structure file size (tens of rows), the extra queries validateRow() already makes are not a real performance concern. Have `commitRow()` ignore the passed `tx` and let `createSchoolClassWithSections()` open its own transaction as before — rejected; traced through the retry path and found a real idempotency bug (a row that actually succeeded could be marked `FAILED` on a retry after a partial failure), not just a theoretical inelegance. Keep `SCHOOL_CLASS`/`SECTION`/`SUBJECT` as three separate `ImportEntityType` values alongside the new `ACADEMIC_STRUCTURE` — rejected; confirmed zero real usage via grep, so correcting the one-sprint-old mistake now avoids permanent unused clutter for no compatibility benefit. Deduplicate raw rows within `validateRow()` (checking sibling rows via a factory closure) rather than at the grouping stage — rejected in favor of deduplicating during grouping itself, once it became clear grouping already guarantees uniqueness structurally, making a second, redundant duplicate check in the validator dead code.

## D-046 — Sprint D4 Import Engine Freeze & Architecture Certification: 16-Question Certification Against Real Code (Not Assumed), One Genuine Gap Found and Fixed (`student`/`teacher`/`attendance.service.ts` Gain the Same `tx` Passthrough D2 Gave `academic.service.ts`, Verified Live With a Real Rollback Test), Two Narrower Composability Constraints Named But Deliberately Not Fixed, Epic D's Core Engine Declared Frozen

- **Description:** A freeze-certification sprint, not a feature sprint — the explicit instruction was to challenge every architectural decision made across D1–D3 against real code, not assume it still holds, and fix only what genuinely needs fixing.

  **The one genuine gap, found by direct inspection, not assumption:** D-044's own "Future Review Required" note predicted that the transaction-passthrough gap it fixed for `academic.service.ts` "almost certainly exists in `student.service.ts`/`teacher.service.ts`/`attendance.service.ts` too." Direct inspection confirmed it: zero of the three had a `tx?:` parameter. `registerStudent()`, `enrollStudent()`, `registerTeacher()`, `deactivateTeacher()` are structurally identical to D2's already-fixed `createSchoolClassWithSections()`/`createAcademicSubject()` (return their DTO from inside their own transaction, no separate read) and got the identical `run(t)`-closure fix.

  A second, deeper wrinkle surfaced that D-044 hadn't anticipated: `assignTeacher()`, `updateTeacherAssignment()`, `openAttendanceSession()`, `markAttendance()`, `submitAttendance()` all do a _separate, standalone read_ after their own transaction commits (`findTeacherAssignmentById()`/`findAttendanceSessionById()`/`findAttendanceRecordById()`), explicitly commented as deliberate — avoiding Prisma's `create()+include` query-decomposition-inside-an-open-transaction bug (a real, previously-found issue, not theoretical). Tracing the actual repository code (`teacherAssignment.repository.ts`, `attendanceSession.repository.ts`, `attendanceRecord.repository.ts`) showed the restriction is narrower than the comments implied: it's specifically `create()+include` in the _same_ call that decomposes; a plain `findUnique()+include`, issued separately, is a single non-decomposed query and is safe _inside_ a still-open transaction. **Fixed**: the three `findXById()` repository functions gained an optional `tx` parameter (defaulting to `db`, so every existing call site is unaffected), and the five service functions above now read back their just-created/updated row via the _same_ transaction instead of waiting for commit — removing the "standalone read" entirely, not just making it composable.

  **Verified live**, not by typecheck alone (a scratch script, cleaned up after itself, no fixtures left behind): (1) `registerStudent()` called with an externally-provided `tx`, forced to roll back — confirmed the student did _not_ persist, proving the write genuinely landed in the caller's own transaction rather than a second, independent one D-044 already showed was a real idempotency risk. ## D-049 — Sprint E2 Attendance Operations Workspace: The First Real Teacher-Facing Daily Workflow — Attendance Home, Session Grid, and History, Composed Entirely From Sprint 5's Own Unmodified `submitAttendance()`/`openAttendanceSession()`, Session Locking Implemented as Derived State (No Schema Change), a User-Clarified "Late" vs. `LEAVE` Wording Question Resolved Without Touching the Existing Enum

- **Description:** The first genuinely daily, teacher-facing workflow this platform has shipped — everything before this sprint (Sprints D/E0/E1) was either Admin-facing or pure architecture certification. A Business Workflow Review ("every morning, a teacher opens the system and needs to mark attendance for today's class in under 2 minutes") shaped the three-screen design: Attendance Home (get to today's section fast) → Attendance Session (mark the whole class in one screen, no per-student round trips) → a Summary shown immediately on submit, with History as a separate, explicitly read-only look-back screen.

  **A genuine wording ambiguity, resolved by asking rather than guessing:** this sprint's own task prompt specified marking "Present / Absent / Late / Half Day," but the actual `AttendanceStatus` enum (built Sprint 5, in daily use, documented in `WORKFLOWS.md § 2`/`BUSINESS_RULES.md § 4`/`DATA_DICTIONARY.md`) is `PRESENT | ABSENT | HALF_DAY | LEAVE` — no `LATE` value exists anywhere in the domain docs, a 4-for-4 mismatch on the 4th slot. Per `AI_RULES.md § 3`'s own "ask when uncertain... this project favors correctness over speed" and this sprint's own "DO NOT redesign Attendance architecture unless a genuine implementation contradiction is discovered," the Project Owner was asked directly rather than either silently adding a new enum value (a real migration to a live table) or silently mislabeling `LEAVE` as "Late" in the UI (misrepresenting what's actually stored). **Resolved: keep the existing four values and their existing labels (`Leave`, not `Late`) untouched** — the task's own wording was imprecise, not a deliberate new requirement. No migration, no `BUSINESS_RULES.md`/`DATA_DICTIONARY.md` change.

  **A second, structural gap found and resolved without a schema change: session "locking."** This sprint's own instruction ("after Final Submit, Session becomes locked... provide extension point for future permission override") has no direct schema support — `AttendanceSession` has no status/lock column (`reopenAttendance()`'s own existing comment already states this: "there is no separate open/closed status on `AttendanceSession` to toggle," `DOMAIN_MODEL.md § 7.1`). Rather than add one, "locked" is **derived**: a session is locked once every currently-enrolled student in its section has an `AttendanceRecord` (`markedCount === totalStudents && totalStudents > 0`), computed fresh on every read in `attendanceSessionWorkspace.service.ts`, never stored. This cleanly distinguishes an in-progress/partial submission (resumable) from a complete one (locked) with zero migration risk to a live table, and composes naturally with `submitAttendance()`'s own existing idempotent-upsert design — a partial submit today, a completing submit tomorrow, both just ordinary calls to the same unchanged function. The "future permission override" extension point is `canOverrideAttendanceLock()` (`src/lib/authorization/permissions.ts`, Admin-only, not wired to any UI control this sprint) — `submitAttendanceGrid()` checks it before rejecting a submit against an already-locked session, live-verified to both correctly reject without it and correctly succeed with it.

  **Zero new database tables, zero changes to `submitAttendance()`/`openAttendanceSession()`/`markAttendance()`/`reopenAttendance()`** (`src/services/attendance/attendance.service.ts`, unmodified since D-046) — every new capability this sprint adds is a _composition_ over those four functions, per this sprint's own explicit "reuse Attendance services, do not bypass them, only compose" instruction, confirmed by `grep`: no new service or Server Action in this sprint calls `upsertAttendanceRecord()`/`createAttendanceSession()` directly. `submitAttendance()`'s own transaction shape — one transaction per submission, `AttendanceRecord × N` upsert + `AuditLog × (N+1)` — was reviewed against `TRANSACTION_BOUNDARIES.md § 2`'s "Daily attendance submission" row and confirmed unchanged and still correct; this sprint's own lock check runs entirely _before_ that transaction opens (a plain read), never inside it.

  **The interactive Attendance Grid** (`src/app/teacher/attendance/[sectionId]/AttendanceGrid.tsx`) is this codebase's first Client Component — search, sort, per-student status buttons, Bulk Present/Bulk Absent, and Reset Session are all local `useState`, zero server round trips; the one real Server Action (`submitAttendanceGridAction`) is called directly (not via a native `<form>`, since the payload is a structured array `AttendanceRecord[]`, not `FormData`-shaped) and follows the exact success-`redirect()`-outside-`try/catch` shape D-045 already established as this framework's own recurring correct pattern. "Reset Session" is deliberately client-side-only, never a server call — `AttendanceRecord` has no delete mechanism by design (`DOMAIN_MODEL.md § 7.2`, corrections are audited edits, never deletions), so a true server-side "wipe" would have contradicted that existing, correct discipline; the client-side interpretation ("clear my current, not-yet-submitted selections") needed no new capability at all.

  **Authorization scoped to Class Teacher, not every subject teacher** — `PERMISSION_MATRIX.md § 5`'s "Teacher: CRU, assigned sections only" doesn't disambiguate which `TeacherAssignment` grants attendance rights; this sprint reads it as `isClassTeacher = true` specifically, matching real school practice (daily attendance is a homeroom responsibility) — a deliberate, named scoping decision, not an oversight, enforced server-side in `assertCanTakeAttendanceForSection()` (checked on every read _and_ every submit, never trusted from a disabled UI element alone).

  **Verified live**, not by typecheck/build alone: a real seeded Class Teacher (Meera Joshi, Class 1 - A) accessed via a minted session showed the correct "Not started today — 0/1 marked" state on Attendance Home; a direct call to `submitAttendanceGrid()` (mirroring exactly what the Grid's own Submit button sends) correctly transitioned the session from unlocked (0/1) to locked (1/1, 100% present); a second submit attempt without override correctly threw `AttendanceLockedError`; the identical submit _with_ `canOverrideLock: true` correctly succeeded, proving the extension point genuinely works, not just compiles; the re-rendered Session page correctly showed the "fully marked and locked" message; History correctly listed the session with the right class/section/teacher name; and — the specific integration this sprint's own instruction asked to verify without redesigning — the Student Workspace's existing Attendance Summary (D-048, unmodified) correctly showed "100%, Present: 1" immediately after the submission, with zero code changes to Student Workspace itself. A nonexistent section 404s; an Admin session (wrong role for this route entirely) redirects at the existing Teacher layout guard; no session redirects to `/login`.

- **Reason:** "Focus on real school operations, not CRUD," per this sprint's own instruction — the value here is a genuinely fast, correct daily workflow built on top of already-correct, already-tested primitives (Sprint 5's own `submitAttendance()`), not a new abstraction layer. Resolving the Late/Leave wording question by asking, rather than either silently changing a live enum or silently mislabeling stored data, is exactly the discipline `AI_RULES.md` names as this project's own priority over speed. Deriving "locked" instead of storing it is the same "don't build ahead of a concrete need" discipline this project already applied in D-029/D-032/D-047 — a stored lock flag would be real, permanent schema surface for a distinction the data already implies.
- **Alternatives Considered:** Silently relabel the existing `LEAVE` status as "Late" in the UI to match the task's own wording exactly — rejected by the Project Owner's own explicit choice; misrepresents what's actually stored (a student on approved leave is not the same fact as a student who arrived late), and this project's own established discipline (D-047, D-048) treats "the task's wording vs. the real domain model" mismatches as things to resolve honestly, not paper over. Add a real `LATE` enum value and a `status`/`isLocked` column to `AttendanceSession` — rejected; both are genuine, non-cosmetic schema changes to live tables this sprint's own "DO NOT redesign... unless a genuine implementation contradiction" instruction explicitly gates, and neither was a contradiction once the derived-state alternative was found to work cleanly. Auto-save each status click as its own `markAttendance()` call (real-time persistence) instead of one batch Submit — rejected; 40 individual round trips per section is slower and more failure-prone than one atomic batch, and contradicts `TRANSACTION_BOUNDARIES.md § 2`'s own "Daily attendance submission" being modeled as one transaction, not N. Give "Reset Session" a real server-side delete path for already-submitted records — rejected; `AttendanceRecord` has no delete mechanism by design, and the client-side-only interpretation already satisfies the actual named need (clearing an in-progress, not-yet-submitted draft) without touching that discipline.
