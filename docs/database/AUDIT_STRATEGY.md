# Audit Strategy

**Purpose:** How `AuditLog` — already conceptually defined in [DOMAIN_MODEL.md § 9.1](../domain/DOMAIN_MODEL.md#91-auditlog) as append-only, no delete path, no `updatedAt` — is physically realized: enforced (not just documented) as append-only, sized correctly for a 15-year horizon, and handled correctly given it can contain PII by inheritance ([DATA_DICTIONARY.md](../domain/DATA_DICTIONARY.md)'s own note).

---

## 1. Why This Table Gets Its Own Document

`AuditLog` is the one table every other mutation in the entire schema writes to — its physical design mistakes don't stay contained to one feature, they compound across all 29 entities. It's also, per [PERFORMANCE_GUIDELINES.md § 2](./PERFORMANCE_GUIDELINES.md#2-growth-projections), on track to be the single largest table in the schema at real scale — larger even than `AttendanceRecord`, since it captures every mutation across every entity, not just one workflow's. Getting its physical design right before the first row is written is exactly the "cheap now, expensive later" case this whole review exists to catch.

## 2. Enforcing Append-Only at the Database Layer, Not Just the Application Layer

[DOMAIN_MODEL.md § 9.1](../domain/DOMAIN_MODEL.md#91-auditlog) already states "no delete path at all, even for Admin" — correctly, as a business rule. Physically, this should not rest on "the application code simply never calls `.update()` or `.delete()` on this model." Recommend:

```sql
REVOKE UPDATE, DELETE ON "AuditLog" FROM app_role;
GRANT INSERT, SELECT ON "AuditLog" TO app_role;
```

Where `app_role` is the Postgres role the application's connection string actually authenticates as. This is defense in depth: a bug, a compromised credential, or a future engineer unfamiliar with this rule attempting a "cleanup" script would all be physically blocked at the database layer, not just discouraged by convention. This is the same reasoning already applied to `TransferCertificate` in [DATABASE_REVIEW.md § 12](./DATABASE_REVIEW.md#12-transfer-certificate) — recommend the identical grant-revocation there too, and on `PromotionRecord` and `ReportCard` per the refinements in [DATABASE_REVIEW.md § 13](./DATABASE_REVIEW.md#13-promotion). A migrations-runner role (used only by CI/CD deploy, never by the running application) needs broader grants to alter these tables' _schema_ — the revocation above applies to the application's runtime role specifically, not the migration role.

## 3. Partitioning

**Recommendation: partition `AuditLog` by range on `timestamp` (monthly or quarterly) from its very first migration**, even though V1 data volume doesn't demand it on day one. The argument is the same one already established for `schoolId`-from-day-one ([PRODUCT_ARCHITECTURE.md § 2](../PRODUCT_ARCHITECTURE.md#2-future-architecture)): partitioning an empty or near-empty table costs almost nothing to set up correctly; converting an unpartitioned multi-million-row table into a partitioned one later requires either a maintenance window, a logical-replication-based online migration, or a third-party tool most teams evaluating this system won't have already in hand.

**The honest caveat, stated plainly rather than glossed over:** Prisma's native support for declarative partitioning is limited — partitioned tables typically need to be created and managed via raw SQL in a migration's `.sql` file rather than pure `schema.prisma` syntax, and Prisma's own model of "one table = one model" doesn't map cleanly onto "one logical table backed by N physical partitions." This is real added complexity for whoever implements Migration 000 (see [MIGRATION_PLAN.md](./MIGRATION_PLAN.md)), not a free win. **If the implementing team judges that complexity isn't worth taking on before there's a second real school generating real audit volume, the acceptable fallback is: create `AuditLog` unpartitioned now, but never add a composite primary key or unique constraint that would block converting it to a partitioned table later** (Postgres requires the partition key to be part of any unique constraint on a partitioned table) — i.e., keep the schema partition-_ready_ even if partitioning itself is deferred. This is presented as a genuine judgment call, not a mandate — see [README.md § 5](./README.md#5-self-review) for why this review avoids over-mandating scale the system doesn't have yet.

## 4. Why `AttendanceRecord`/`MarksRecord` Are Mutable, Not Event-Sourced

Worth stating explicitly since it's the one place this review recommends _against_ treating a fact table as strictly immutable, in apparent tension with everything else in this document. A fully event-sourced design would make `AttendanceRecord`/`MarksRecord` themselves append-only (every correction is a new row; "current state" is always "the latest row for this key"), with `AuditLog` becoming redundant for these two tables specifically. Considered and rejected for this system: event-sourcing every attendance mark adds real query complexity (every read needs a "latest per key" window function or a materialized view) for a correction frequency that's genuinely low (a teacher fixing a mis-marked student, not a high-frequency operation) — the mutable-row-plus-`AuditLog` pattern already gives complete traceability (who changed what, from what, to what, when) without that complexity. This is a deliberate scale judgment, not an oversight: revisit only if a real requirement emerges for point-in-time reporting ("what did this report look like as of the exact moment it was generated, even after later corrections") that the current design can't serve — nothing in [REPORTING_MODEL.md](../domain/REPORTING_MODEL.md) currently asks for that.

## 5. Retention

No archival _tooling_ is in scope for V1 — but the retention _policy_ should be decided now, in writing, rather than left as an unstated assumption that accumulates risk silently for a decade. Recommend: full-fidelity retention for at least 7 years (a common floor for Indian financial/educational record-keeping expectations — this is a policy input for School Admin/legal counsel to confirm, not an engineering-only decision, flagged the same way [BUSINESS_RULES.md § 8](../domain/BUSINESS_RULES.md#8-data-protection-dpdp-act-2023) flags DPDP consent capture as a real gap rather than silently assuming an answer). Once partitioning (§ 3) is in place, "archive" becomes a mechanical operation — detach an old partition, compress it, move it to cheaper storage — rather than a bespoke deletion job; this is the concrete payoff of partitioning early that justifies its setup cost.

## 6. PII Handling in `beforeValue`/`afterValue`

[DATA_DICTIONARY.md § 7](../domain/DATA_DICTIONARY.md#7-cross-cutting-auditlog) already flags that these JSON columns inherit the PII classification of whatever they're logging — a mutation touching `RteDetails.incomeCertificateDocumentId` produces an audit row that's Sensitive PII by inheritance, even though `AuditLog`'s own schema looks generic. Physical recommendation: **for a short, explicitly named list of the most sensitive fields** (income/category certificate document references, any future field holding a raw document number rather than a reference ID), the mutation's audit-logging code should redact the actual value in `beforeValue`/`afterValue` (store `"[REDACTED]"` or a one-way hash instead of the real value) while still recording _that_ a change occurred and _who_ made it. This preserves the audit trail's core purpose (accountability, not necessarily full content reproduction) without turning `AuditLog` into a second, less-access-controlled copy of the schema's most sensitive data. This is an application-layer implementation detail, not a schema change — flagged here because the schema review is what surfaced the risk, and it would be easy for whoever implements audit-logging generically (e.g., a Prisma middleware that snapshots every mutation automatically) to miss it precisely because it's _not_ visible in the schema itself.

## 7. Access Control on Read

`AuditLog` should be readable only by Admin, per [PERMISSION_MATRIX.md § 7](../domain/PERMISSION_MATRIX.md#7-system--cross-cutting) — "R (read-only, never editable by anyone)," Admin-only. Worth restating here because it's easy, once `AuditLog` exists as a genuinely useful debugging tool, for it to informally become something engineers query directly in production for troubleshooting — any such access must go through the same role-scoped query layer as every other read in this system ([ARCHITECTURE.md § 7](../ARCHITECTURE.md#7-security-principles)), not a direct database connection that bypasses application-level access control, precisely because of § 6's PII inheritance.
