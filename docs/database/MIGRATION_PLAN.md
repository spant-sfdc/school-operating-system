# Migration Plan

**Purpose:** Small, independent, ordered migrations for Epic B's actual implementation to follow — no mega migration that creates all ~29 tables in one shot. Builds on [docs/domain/DOMAIN_MODEL.md § 11](../domain/DOMAIN_MODEL.md#11-recommended-build-order), reordered here based on _physical_ dependencies (what must exist in the database for the next migration to be valid) rather than the domain model's _conceptual_ build order (what makes sense to design first) — the two orders mostly agree, with one deliberate change explained in § 1.

---

## 1. The One Reordering: `AuditLog` Moves First, Not Last

[DOMAIN_MODEL.md § 11](../domain/DOMAIN_MODEL.md#11-recommended-build-order) lists `AuditLog` last, "only because it's cross-cutting infrastructure, not a milestone of its own" — a reasonable conceptual ordering, but a physical review should push back: every migration after the first one will want to write audit entries for the mutations it enables. Shipping five migrations' worth of features before `AuditLog` exists means those features either ship without audit coverage temporarily, or need a retrofit pass once `AuditLog` finally arrives — both worse than just building the (structurally trivial, dependency-free) audit table first. **`AuditLog` is Migration 000, before anything else.**

## 3. Why This Order, Not a Single Migration

Three concrete reasons, not just "small migrations are good practice" as an abstract principle:

1. **Each migration is independently testable in a review.** A reviewer can read Migration 003 (Identity) and verify it's correct without having to hold the entire ~29-table schema in their head at once — directly useful given this same schema will likely be implemented across more than one PR/session, the same way the Website Engine was built page-by-page rather than all at once.
2. **Rollback blast radius stays small.** If Migration 006 (Attendance) reveals a mistake after a few weeks of real data, rolling it back doesn't also roll back Identity, People, or Academic Structure, which have nothing to do with the mistake.
3. **It matches the actual order features will ship in.** Per [DOMAIN_MODEL.md § 11](../domain/DOMAIN_MODEL.md#11-recommended-build-order)'s own reasoning (reaffirmed, not contradicted, by this reordering): Epic B should ship Student/Teacher management before Attendance is buildable at all, since attendance needs real enrollments to mark against. A migration plan that mirrors the feature-shipping order means the database schema is never "ahead of" what the application actually uses yet — no speculative tables sitting empty for months.

## 4. What "Small and Independent" Does Not Mean

Not one-table-per-migration. `AttendanceSession` and `AttendanceRecord` ship together (006) because neither is independently useful — an `AttendanceSession` with no `AttendanceRecord`s serves no product purpose, and vice versa is structurally impossible (an `AttendanceRecord` requires a `sessionId`). The right granularity is **one coherent, independently-shippable capability per migration**, not an arbitrary table count — the same "genuinely distinct feature" judgment [DEVELOPMENT_CONVENTIONS.md § 2](../DEVELOPMENT_CONVENTIONS.md#2-directory-organization) already applies to component promotion, applied here to schema changes instead.

## 5. Reversibility

Every migration in this plan should ship with a working `down` migration (Prisma generates migration history but reversal is the implementer's responsibility to verify, not automatic) **except** 000 and any migration touching `AuditLog`, `TransferCertificate`, or `PromotionRecord` once real data exists in them — reversing a migration that would drop an append-only historical table is exactly the kind of destructive operation this project's own git-safety discipline ([system rules on destructive operations](../AI_RULES.md)) already treats with heightened caution, extended here to schema operations. Recommend: these three tables' migrations are reversible _before_ they hold real data (early in the project), and treated as forward-only once real records exist — a policy to state explicitly in the migration's own commit message when that line is crossed, not something to infer later.

## 6. Sequencing Relative to Epic B's Own Roadmap

Cross-referencing [ROADMAP_V2.md § Epic B](../ROADMAP_V2.md#epic-b--administration)'s own milestone list: "First Prisma models (Student, Teacher)... with `schoolId` scoping from the first migration" maps to Migrations 001–004 above, not a single combined step — Epic B's roadmap entry was intentionally written at product-milestone granularity, not migration granularity; this document is the refinement of that one roadmap line into an actual sequence, not a change to the roadmap's own scope or ordering.
