# Transaction Boundaries

**Purpose:** Which multi-statement writes must be wrapped in a single database transaction, and what the correct behavior is on partial failure. Found missing during the pre-Prisma consistency verification — no document in [docs/domain/](../domain/README.md) or elsewhere in [docs/database/](./README.md) systematically addressed this before now; [docs/domain/EVENT_MODEL.md](../domain/EVENT_MODEL.md)'s single parenthetical aside ("Creates `Student` + `Enrollment` (same transaction)") was the only existing mention, and it doesn't generalize to the rest of the schema's own multi-table writes.

---

## 1. The Core Principle

**Any application mutation that writes to more than one table must be wrapped in a single database transaction (Prisma's `$transaction`).** This includes, critically, the write and its own `AuditLog` entry — not just writes that touch two "business" tables. A mutation that succeeds but fails to log itself, or logs itself but fails to actually commit, is a genuine data-integrity problem: the first case produces an unaudited change ([AUDIT_STRATEGY.md](./AUDIT_STRATEGY.md)'s entire purpose defeated silently), the second produces an audit entry describing something that never actually happened.

**The correct behavior on partial failure is always full rollback, never a partial save.** A user-facing operation either completely succeeds or completely fails — an attendance submission that saves 25 of 40 students' records because the 26th hit a constraint violation is worse than one that saves none, because the failure isn't visible: the teacher sees "done" and the class register is silently wrong for the rest of the term.

**Single-table writes still need the transaction pattern** whenever `AuditLog` is involved — which, per [AUDIT_STRATEGY.md § 2](./AUDIT_STRATEGY.md#2-enforcing-append-only-at-the-database-layer-not-just-the-application-layer), is every mutation in the system. A `Guardian` contact-detail edit is a "single-table" business operation but a two-table transaction (`Guardian` + `AuditLog`) in practice. This document's table above lists only the genuinely multi-_business_-table cases; the `AuditLog` pairing applies universally on top of it.

## 3. Batch Size — Why a 40-Row Transaction Is Fine, and When That Stops Being True

Attendance submission for a full section (up to ~40 students, per [PRODUCT_VISION.md § 3](../PRODUCT_VISION.md#3-target-audience)'s upper bound) writing 40 `AttendanceRecord` rows plus 40 `AuditLog` rows in one transaction is entirely appropriate at this scale — tens of rows, held for milliseconds, is not meaningful lock contention on any Postgres instance sized for this workload. **No special batching, chunking, or queue-based processing is needed for V1.** This is worth stating explicitly precisely because it's tempting to over-engineer: "batch writes" sounds like it should need infrastructure, and at this row count, it doesn't. Revisit only if a future feature needs a genuinely large single write (thousands of rows) — nothing in the current domain model does.

## 4. The One Exception: Academic Year Rollover Should _Not_ Be One Transaction

[WORKFLOWS.md § 8](../domain/WORKFLOWS.md#8-academic-structure-setup-workflow-start-of-each-academic-year) already flags this workflow as needing its own treatment ("worth Epic B treating as its own late-stage milestone") — this document adds the concrete physical reason. Rolling over an entire school (hundreds to thousands of students, per § 2's `Enrollment` volume) means potentially thousands of `PromotionRecord` + `Enrollment` + `AuditLog` writes in one operation. Wrapping all of it in a single database transaction would hold row locks across a large fraction of the school's data for however long that batch takes to process — real contention risk against any concurrent read (a parent checking last year's report card while rollover is running), and a single failure partway through would roll back potentially hundreds of already-correct decisions along with the one that failed.

**Recommendation:** process rollover as a supervised, resumable batch — one transaction per student (or per small batch of students, e.g., per section), with its own progress-tracking record (`academicYearId` + which students have been rolled over so far) so a failure or interruption partway through doesn't require restarting from zero and doesn't hold a school-wide lock for the operation's full duration. This progress-tracking mechanism is itself a real, if small, design decision Epic B's rollover implementation will need to make explicitly — named here as a requirement, not designed further, consistent with this review's own scope boundary (see [README.md § 2](./README.md#2-what-this-is-not)).

## 5. Concurrency: Check-Then-Act Races

[WORKFLOWS.md § 2](../domain/WORKFLOWS.md#2-daily-attendance-workflow)'s own "does today's session exist?" step, read literally, is a check-then-create pattern — if two requests for the same section+date arrive concurrently (unlikely for one teacher, but not impossible with a flaky client retry), both could pass the "doesn't exist" check before either commits, and the second insert would fail against the `(sectionId, date)` unique constraint. **This is the correct outcome, not a bug to design around** — the unique constraint is the actual concurrency guard; the "check first" step in the workflow diagram is a UX optimization (avoid an unnecessary round-trip for the common case), not the mechanism that guarantees correctness. Recommend implementing this as an upsert (`ON CONFLICT (sectionId, date) DO NOTHING` or `DO UPDATE`, Prisma's `upsert`) rather than a literal `SELECT` then `INSERT`, so the race resolves cleanly at the database layer regardless of request ordering. The same reasoning applies to every other unique-constrained write in this schema that a workflow diagram describes as "check, then create" — `Enrollment`, `MarksRecord`, `TeacherAssignment` — the unique constraint, not the application-level check, is what actually prevents the duplicate.

## 6. What This Document Does Not Cover

Retry policy for a failed transaction (exponential backoff, how many attempts, user-facing messaging on final failure) is an application/UX concern, not a database-physical-design one — out of scope here, same boundary [PERFORMANCE_GUIDELINES.md § 5](./PERFORMANCE_GUIDELINES.md#5-n1-avoidance) already draws between schema-level and code-quality-level guidance.
