# Implementation Log

**Purpose:** The engineering diary for this project. [CHANGELOG.md](./CHANGELOG.md) records _what_ shipped, for anyone; this document records _why_, for future engineers and AI assistants who need the reasoning, not just the result — architecture decisions in progress, rejected ideas, lessons learned, manual review notes, and things to reconsider later.

This is not a duplicate of [DECISIONS.md](./DECISIONS.md). DECISIONS.md holds the final, approved outcome of a decision in a fixed format. This document holds the messier narrative around it — what was tried, what didn't work, what almost happened instead.

## 2026-07-17 — Phase 0A: Initial Documentation Foundation

**Context:** Project started from zero — no code, no docs, no repository history. The instruction set was explicit that no application code should exist yet; the entire deliverable was a documentation foundation.

**What was done:** Authored the initial ten-document set (`PROJECT_CONTEXT.md`, `PRODUCT_REQUIREMENTS.md`, `ARCHITECTURE.md`, `UI_DESIGN_SYSTEM.md`, `AI_RULES.md`, `DECISIONS.md`, `ROADMAP.md`, `TASKS.md`, `CHANGELOG.md`, `README.md`).

**Lessons learned:** Writing all ten documents independently in one pass produced natural, unplanned duplication — tech stack, folder structure, and role definitions ended up stated near-identically in three or four places (`PROJECT_CONTEXT.md`, `ARCHITECTURE.md`, `README.md`). This wasn't a mistake in any single document — each one made sense to be self-contained in isolation — but in aggregate it created multiple sources of truth that could silently drift apart. This directly motivated Phase 0A.1 below.

## 2026-07-17 — Phase 0A.1: Documentation Consolidation & AI Operating System

**Context:** With the initial ten documents in place, the project needed to become a genuine "AI development operating system" — maintainable by another AI or developer six months from now, from documentation alone, with no chat history required.

**Problem identified:** Three categories of issue in the Phase 0A document set:

1. **Duplication** — tech stack, folder structure, role definitions, and design principles were each stated fully in 2–3 places, with real risk of drifting inconsistent over time.
2. **Missing operating layer** — the original ten documents described the _product_ well but had no equivalent for _how an AI or engineer should operate day to day_: no naming conventions, no component registry, no route registry, no reusable implementation prompt, no completion checklist beyond a 7-line summary.
3. **Format debt** — `DECISIONS.md` entries lacked fields (approver, alternatives considered) that matter once more than one person is involved; `CHANGELOG.md` used an ad hoc format instead of an industry-standard one.

**Decision:** Make `PROJECT_CONTEXT.md` the single source of truth, with every other document either (a) holding detail that would bloat `PROJECT_CONTEXT.md` if inlined, cross-linked from it, or (b) a genuinely new operating document (guardrails, conventions, inventories, master prompt, this log) that didn't exist before. See [DECISIONS.md § D-006](./DECISIONS.md#d-006--documentation-consolidation-into-single-source-of-truth).

**Alternative considered and rejected:** Keep every document fully self-contained (accept some duplication as the cost of each document being independently readable). Rejected because the explicit goal — six-months-later maintainability with no chat history — depends on there being exactly one place to update a fact when it changes; duplicated facts are duplicated maintenance burden and a guaranteed eventual inconsistency.

**Alternative considered and rejected:** Merge everything into one enormous `PROJECT_CONTEXT.md` instead of a hub-and-spoke model. Rejected because a single monolithic document becomes expensive to load in full for every task (token cost) and hard to scan; the hub-and-spoke model (one source of truth + focused references) keeps each read scoped to what's actually needed.

**What changed:**

- `PROJECT_CONTEXT.md` restructured around Current Phase/Sprint, Architecture Summary, Roles, Completed/Pending Features, Folder Structure, Tech Stack, Design Summary, Coding Standards Summary, Decisions, Open Questions, Risks, and a full Reference Links map.
- New documents created: `PROJECT_GUARDRAILS.md`, `DEVELOPMENT_CONVENTIONS.md`, `COMPONENT_INVENTORY.md`, `ROUTES.md`, `FEATURE_STATUS.md`, `DEFINITION_OF_DONE.md`, `MASTER_PROMPT.md`, this log.
- `AI_RULES.md` expanded with concrete engineering-discipline rules (minimal diffs, no placeholder code, no `console.log`, manual verification requirement) that were previously implied but not stated.
- `DECISIONS.md` reformatted to a richer template (ID, Title, Description, Reason, Alternatives Considered, Approved By, Date, Status, Affected Documents, Future Review Required).
- `CHANGELOG.md` converted to Keep a Changelog format.
- `README.md` trimmed of duplicated tech stack/folder structure detail in favor of links to `PROJECT_CONTEXT.md`/`ARCHITECTURE.md`, and expanded with a documentation map and development workflow section.

**Future consideration:** Once Phase 0B produces real code, verify that `COMPONENT_INVENTORY.md` and `ROUTES.md` are actually kept current in practice, not just in principle — an inventory nobody updates is worse than no inventory, because it creates false confidence. Revisit whether a lint rule or PR template checklist item can enforce the documentation-update step in [DEFINITION_OF_DONE.md](./DEFINITION_OF_DONE.md) mechanically rather than relying on discipline alone.

## 2026-07-17 — Phase 0B.1: Project Scaffolding

**Context:** First implementation phase — bootstrap the actual Next.js repository per the approved architecture and design system documents, infrastructure only, no business functionality.

**Environment problems encountered and resolved:**

- The system's `pnpm` was a corepack shim pointing at Node's bundled corepack, which failed with a signature-verification error (`Cannot find matching keyid`) — a known corepack issue unrelated to this project. Resolved by removing the broken shims and installing `pnpm` directly via `npm install -g pnpm`, pinned to the 9.x line (`pnpm@11` required `node:sqlite`, not available on this Node build).
- Prisma's CLI refused to install on the local Node v23.1.0 (an odd-numbered "Current" release; Prisma only supports 20.19+, 22.12+, 24.0+). Rather than changing the developer's global Node version, installed Node 22 LTS via Homebrew as a keg-only formula (does not touch the global `node` symlink) and ran only the Prisma-specific commands with it prepended to `PATH` for that command. This is documented as a current risk in `PROJECT_CONTEXT.md` — the next contributor needs the same workaround until a project Node version is formally pinned (e.g., via `.nvmrc`).

**Decision — `create-next-app@latest` vs. a pinned version:** `create-next-app@latest` scaffolded Next.js 16, not the approved Next.js 15. `create-next-app` is versioned in lockstep with `next` itself, so re-running with `create-next-app@15` scaffolds the correct major version directly — cleaner than scaffolding latest and downgrading after the fact, which would leave Next-16-flavored generated config (e.g., default Turbopack dev config) in a Next-15 project.

**Decision — `src/` layout vs. the originally documented root-level structure:** The Phase 0B.1 instructions explicitly requested a `src/`-based feature-first layout, but `ARCHITECTURE.md`'s original folder tree put `app/`, `components/`, `lib/`, `types/` at the repository root. Rather than silently picking one, treated this as a legitimate architecture update: kept every internal composition decision (route groups, component categories, lib organization) unchanged, wrapped them in `src/`, and left `prisma/`, `public/`, `docs/`, and config files at root (the standard split, and `prisma/` in particular is conventionally always root-level regardless of `src/` usage). Recorded as [DECISIONS.md § D-007](./DECISIONS.md#d-007--src-directory-layout) and updated `ARCHITECTURE.md`'s tree to match, rather than leaving the documentation and the repository disagreeing.

**Decision — what "installation only" means for Auth.js and Prisma:** The task explicitly marked Auth.js and Prisma as "installation only." For Prisma, ran `prisma init` (datasource + generator blocks, zero models) and validated `prisma generate` succeeds — this is the mechanical, standard output of "installing Prisma," not a schema decision. For Auth.js, installed the `next-auth@beta` (v5) package only — deliberately did _not_ create `src/lib/auth.ts` with providers/session config, since that would be actual auth logic, explicitly forbidden. Also deliberately did **not** create `src/lib/db.ts` (a Prisma client singleton), even though `ARCHITECTURE.md` documents it as expected `lib/` content, because instantiating and successfully importing `PrismaClient` meaningfully only matters once there's a schema to query against — creating it now would be inert boilerplate with no immediate purpose, and the task's own task list (as opposed to the tech-stack list) never asked for it. Deferred to Phase 0B/2, tracked in `DECISIONS.md`'s Rejected/Deferred table.

**Decision — `next-intl` and `next-pwa` installed but not wired:** Both were explicitly named in the approved tech stack, so both were installed. Neither was configured: `next-intl`'s standard setup requires either a `[locale]` route segment (which would restructure the approved route-group layout) or a `request.ts` config that has no locales to serve yet; `next-pwa` ships with zero TypeScript type declarations (confirmed by inspecting `node_modules/next-pwa/package.json` — no `types` field, no `.d.ts` files), so importing it in `next.config.ts` under `strict: true` would either fail type-checking or require an `any`/`ts-expect-error` escape hatch, both disallowed by `AI_RULES.md`. Wiring either now would have meant violating an explicit project rule or making an unrequested architectural change. Left `next.config.ts` at its clean default and recorded the reasoning in [DECISIONS.md § D-008](./DECISIONS.md#d-008--phase-0b1-tooling--dependency-additions) rather than silently skipping the tech-stack instruction.

**Decision — removing the shadcn-generated `Button` component:** `shadcn init` auto-installed a `Button` primitive as a verification artifact of its own init process. The task's "DO NOT CREATE" list explicitly says "No Components." Reasoned that a mechanically-generated example component still counts as a component under a literal reading of that instruction, so removed it after init completed, keeping only `components.json` (config) and `src/lib/utils.ts` (the `cn()` helper, infrastructure rather than a component). First component gets added when Phase 1 implementation actually needs one.

**Decision — replacing the default Next.js demo homepage:** `create-next-app` generates a marketing-style demo page (Next.js logo, "Deploy now" / "Read our docs" links to vercel.com/nextjs.org). Leaving it in place would arguably violate "No Homepage" more than replacing it would — it's a fully designed page with branding and outbound navigation, not neutral scaffolding. Replaced it with a single unstyled sentence confirming the infra is running, used only to prove the dev server and build pipeline work end-to-end. This is explicitly a placeholder for Phase 1, not a homepage design decision.

**Lessons learned:** When a task's own instructions internally disagree (Step 4 says follow `ARCHITECTURE.md` exactly; Step 5 says use a `src/` layout `ARCHITECTURE.md` didn't have), the right move is to update the documentation to match the resolved decision in the same change — not to implement one and leave the other silently stale. Also: "install only" is ambiguous enough on its own that it's worth spelling out, per dependency, exactly what "installed" produced (a package.json entry vs. an initialized config vs. a wired-in feature) rather than assuming the phrase self-explains.

## 2026-07-17 — Phase 0B.1 Finalization: Stabilization Pass

**Context:** Phase 0B.1 scaffolding was functionally complete but had never been reviewed as a whole or committed to git. This session's job was explicitly _not_ to build anything new — review, fix defects, verify the pipeline, pin the runtime, and prepare the repository for its first commit/push/tag.

**What the review found:** Two genuinely orphaned dependencies. The previous session installed `shadcn`'s default `Button` component during `shadcn init`, then deleted it (per the "No Components" instruction) but never checked whether `Button`'s own dependencies (`@base-ui/react`, `class-variance-authority`) were still needed elsewhere. `grep` across `src/` confirmed zero remaining imports of either — genuine dead weight, unlike `next-intl`/`next-pwa` which are unused but explicitly approved tech stack. Also found `shadcn` itself sitting in `dependencies` rather than `devDependencies` — it's a CLI tool invoked via `pnpm exec`/`pnpm dlx`, never imported by application code, so its production classification was simply wrong.

**Decision — pin the runtime, then make the pin true:** The task specified an exact `.nvmrc`/`engines` block (Node `>=22 <23`, pnpm `>=10`). Adding it immediately broke `pnpm install` in the working session (`ERR_PNPM_UNSUPPORTED_ENGINE`) — the session's actual toolchain was pnpm 9.15.9 on Node 23.1.0, a informal workaround from the prior session, not a real match. Rather than weakening the constraint to make the error go away (which would defeat the entire purpose of pinning), upgraded the toolchain to actually satisfy it: confirmed `node:sqlite` works under the already-installed Node 22.23.1 (Homebrew keg-only), then installed pnpm 10.34.5 globally using that Node. This replaced the global `pnpm` binary outright, so the fully-pinned toolchain (Node 22 + pnpm 10) now works with zero warnings, and even the machine's _default_ Node 23 environment now only produces a soft warning (not a hard failure) on `pnpm install`, since pnpm 10's engine check treats the `pnpm`-version mismatch as the hard gate and the `node`-version mismatch as informational. This meaningfully narrows the risk flagged in the previous session's `IMPLEMENTATION_LOG.md`/`PROJECT_CONTEXT.md` entry — it's no longer "the whole toolchain needs a workaround," it's "a from-scratch `prisma` package install specifically needs Node 22, once."

**Decision — pnpm 10's build-script blocking:** Upgrading to pnpm 10 surfaced a new default behavior — postinstall scripts are blocked unless explicitly approved (`Ignored build scripts: prisma@7.8.0`). Ran `pnpm approve-builds --all` rather than ignoring the warning, since Prisma's postinstall fetches its query-engine binary and silently skipping it would produce a passing `pnpm install` that quietly leaves Prisma non-functional later. This wrote `pnpm-workspace.yaml` (`allowBuilds: { prisma: true }`) — a new, legitimate config file that gets committed alongside everything else.

**Decision — merge this pass into the v0.1.0 changelog entry rather than adding v0.1.1:** Since no commit or tag existed yet when this session started, v0.1.0 wasn't actually "released" in any git sense — `CHANGELOG.md`'s own "never edit a past entry" rule exists to protect entries that have already shipped. Folded the dependency cleanup and runtime-pinning changes into the existing `[0.1.0]` entry instead of creating a `[0.1.1]`, since the task's own commit/tag instructions treat this finalization pass as still part of "Phase 0B.1 — Project Scaffolding," not a separate release.

**Lessons learned:** A "finalize and stabilize" pass is exactly the right moment to catch dependency drift from earlier reversals — the kind of thing that's individually invisible (an unused package doesn't fail a build) but accumulates as silent cruft if nobody looks. Also: pinning a runtime constraint is only meaningful if the pass that adds it also verifies the constraint is actually satisfiable in the current environment — an aspirational `engines` field that immediately breaks the local install is worse than no pin at all, because it looks enforced without being tested.

## 2026-07-17 — Phase 1A: Public Website Foundation

**Context:** First real UI work — build the design system, theme tokens, typography, and header/footer/hero chrome every future public page will use. Explicitly not the homepage content itself, not any other page.

**Design tokens gap found:** `UI_DESIGN_SYSTEM.md § 2` names `surface`, `surface-muted`, `success`, `warning`, `info` as required semantic tokens, but the actual `globals.css` shipped in Phase 0B.1 (shadcn's default output) only had `card`, `muted`, `accent`, `destructive` — none of the state-color tokens existed. Added them as new CSS custom properties (light + dark), aliasing `surface`/`surface-muted` to the existing `card`/`muted` values so nothing visually changed for existing usage, while making the semantically-correct token names actually available. Also added `duration-fast/base/slow` and `shadow-soft/elevated` as Tailwind v4 `@theme` entries so "animation duration" and "shadow" are real semantic tokens (`duration-fast`, `shadow-soft` utility classes) rather than raw arbitrary values scattered through components.

**Decision — introduce a real accent color:** `UI_DESIGN_SYSTEM.md` explicitly anticipates this ("neutral palette + one accent for primary, until brand is finalized") and `PROJECT_CONTEXT.md`'s own risk mitigation plan says to revisit before "Phase 1 visual polish" — this is that moment. Picked a deep indigo-blue, recorded as [DECISIONS.md § D-010](./DECISIONS.md#d-010--placeholder-accent-color-for-phase-1a), explicitly flagged as a placeholder swappable at the token layer.

**Decision — `components/website/` folder:** `ARCHITECTURE.md`'s original tree only had `ui/`, `shared/`, `admin/`, `teacher/` — no home for Guest-role composites, even though `COMPONENT_INVENTORY.md` already had a "Website Components" category from Phase 0A.1 anticipating exactly this. `components/shared` is defined as "reusable across roles," which a public header/footer isn't (Admin/Teacher have separate shells) — so filing it there would misrepresent the folder's purpose. Added `components/website/` as a third role-composite folder, parallel to `admin/`/`teacher/`. Recorded as [DECISIONS.md § D-011](./DECISIONS.md#d-011--componentswebsite-folder) and updated `ARCHITECTURE.md`'s tree to match.

**Decision — dependency-free interactivity:** The brief asked for a mobile nav drawer, a theme toggle, and a language switcher, all needing accessible interactive primitives (focus trapping, escape-to-close, popover behavior). The obvious tool would be `@base-ui/react` (shadcn's headless primitive library, already configured in `components.json`) — but it had just been removed as an unused dependency in the Phase 0B.1 finalization pass, and this task explicitly said "do not introduce new dependencies." Rather than re-adding it, hand-built all three with plain React state + Tailwind + the already-installed Framer Motion: `MobileNav` manages its own focus (autofocus first link, Escape closes, body scroll lock), `LanguageSwitch` is a small outside-click-aware popover, `ThemeToggle`/`ThemeProvider` replicate the standard `next-themes` pattern (inline blocking script in `<head>` for no-FOUC, `localStorage` persistence, React Context) without the package. All three are small enough that hand-rolling was less code and less risk than a new dependency would have been.

**Decision — Statistics Strip without fabricated statistics:** The brief explicitly asked for a Hero "statistics strip," but no real school data exists (student counts, founding year, results are all unconfirmed per `PRODUCT_REQUIREMENTS.md § Open Product Questions`). Inventing plausible-looking numbers ("500+ Students") would be exactly the kind of fabrication `CONTENT_GUIDELINES.md § 12` (written earlier in this same session) explicitly forbids — a false claim on a real school's real website. Built the same visual pattern (a horizontal strip of highlighted facts) using qualitative, non-falsifiable pillars instead (Attentive teaching / Considered curriculum / Beyond the classroom / Vidyadhar Nagar, Jaipur) — honest content in the shape the design brief asked for, with the underlying `StatStrip` component fully data-driven and ready to take real numbers once School Admin provides them.

**Bug found via real-browser verification — font silently falling back to Times:** Wired Inter via `next/font/google` and pointed Tailwind's `--font-sans` at it inside `@theme inline`, following the same pattern as the pre-existing (broken, self-referential) `--font-sans: var(--font-sans)` placeholder from Phase 0B.1's scaffold. Build, lint, and typecheck all passed clean — the bug was invisible to every automated check. Screenshotting the running app (headless Chrome, `Page.captureScreenshot` via the DevTools Protocol) showed the headline rendering in a serif font instead of Inter. Root-caused via `getComputedStyle` inspection over a CDP WebSocket connection: `--font-inter` (the CSS variable next/font generates) was defined via a class on `<body>`; `--font-sans: var(--font-inter)` was declared on `:root`/`<html>` — an _ancestor_ of `body`. CSS custom properties only cascade from ancestor to descendant, never the reverse, so `html` could not see a variable defined on its own child, `--font-sans` resolved to nothing, and `font-family` silently fell back to the browser default. Fixed by moving the `next/font` variable classes from `<body>` to `<html>`, which is also the more standard placement for exactly this reason. Verified fixed via the same computed-style inspection (`font-family: Inter, "Inter Fallback"`) and visually via a corrected screenshot.

**Tooling artifact — don't trust `chrome --screenshot` alone for responsive verification:** The initial mobile screenshots (`--headless --window-size=390,844 --screenshot=`) showed what looked like a second, more serious bug: headline and body text overflowing the viewport, clipped mid-word. Chasing this down (CDP `Emulation.setDeviceMetricsOverride` with `mobile: true` at the same 390px width) showed the _actual_ rendered layout was correct — text wrapped cleanly within its container with zero overflow (`h1.scrollWidth === h1.getBoundingClientRect().width`). The CLI `--screenshot` flag doesn't apply mobile viewport/device emulation (no `mobile: true`), so it appears to lay the page out under desktop viewport semantics and then crop the output image to the requested window size, rather than genuinely reflowing at that width — producing a crop that looks exactly like an overflow bug. **Lesson: when verifying responsive layout with headless Chrome, use `Emulation.setDeviceMetricsOverride({ mobile: true, ... })` over a DevTools Protocol connection, not the bare `--screenshot` CLI flag** — the CLI flag is fine for a quick visual gut-check at desktop widths but produced a false positive at mobile width in this session.

**Manual verification performed this session:** Full `pnpm install && pnpm lint && pnpm typecheck && pnpm build` clean. Live dev server checked via headless Chrome + CDP at 390px (mobile, `mobile: true` emulation) and 1440px (desktop), in both light and dark mode (`Emulation.setEmulatedMedia`). Console checked for errors/exceptions via `Runtime.enable` + `Runtime.exceptionThrown` — none found, only the expected React DevTools info message.

**Lessons learned:** A passing build/lint/typecheck proves the code is syntactically and structurally valid — it says nothing about whether the CSS custom property chain actually resolves at runtime in a real browser. This class of bug (silent `var()` fallback due to cascade-direction scoping) is invisible to every tool except actually looking at computed styles in a browser, which is exactly why "start the dev server and use the feature in a browser" is a hard requirement for UI work, not a nice-to-have.

## 2026-07-17 — Phase 1B: Marketing Section Library

**Context:** Build a reusable library of 15 marketing section primitives (`PageHero`, `FeatureGrid`, `Timeline`, `FAQAccordion`, etc.) so every future public page can be assembled without inventing new layout patterns — explicitly the library only, not any actual page.

**Mandatory architecture review — `components/website/` vs. `src/features/`:** The task required reviewing whether `src/components/website/` should remain or move into a suggested `src/features/website/marketing/sections/` structure, framed as "the approved feature-first architecture." No such structure is actually approved anywhere in `/docs` — `ARCHITECTURE.md`'s Component Hierarchy (confirmed current as of the Phase 1A finalization one session prior) is role-segmented (`components/{ui,shared,website,admin,teacher}`), not feature-sliced. Flagging this discrepancy rather than silently complying with the "approved" framing or silently ignoring the suggestion: moving only `website/` to a `features/` tree while Admin/Teacher stayed under `components/` would leave the codebase with two incompatible organizational philosophies for the same kind of problem — worse for six-months-later maintainability than the status quo, not better. Decision: kept `components/website/` exactly where it is (zero existing files touched), and adopted only the genuinely valuable, low-risk part of the suggestion — self-contained per-component folders (`Component.tsx`/`.types.ts`/`.constants.ts`/`README.md`/`index.ts`) — for the _new_ library, added as `components/website/sections/`. Recorded as [DECISIONS.md § D-012](./DECISIONS.md#d-012--marketing-section-library-location-componentswebsitesections-not-srcfeatures). This is the kind of judgment call [AI_RULES.md § Communication Discipline](./AI_RULES.md#3-communication-discipline) exists for: explain the architectural reasoning and flag the conflict, don't just execute the literal suggestion.

**Design choices worth recording:**

- **`ContentContainer` as the one place width/padding lives.** Every section that needs a constrained width (`PageHero`, `ImageText`, `CTASection`, and every demo section in the verification page) composes `ContentContainer` internally rather than repeating `mx-auto max-w-* px-6 lg:px-8`. Avoids the exact kind of duplicated-styling risk the task explicitly warned against.
- **`AnimatedSection` as the one place reveal-on-scroll motion lives.** `FeatureGrid` and `Timeline` are themselves plain Server Components that compose the client `AnimatedSection` around each item (staggered via a `delay` prop) — the correct Next.js Server-Component-composing-Client-Island pattern, and it means only three components in the whole library (`AnimatedSection`, `FAQAccordion`, `StatisticsGrid`) actually need `"use client"`; the other twelve are pure Server Components.
- **`StatisticsGrid` ships with real count-up animation but zero real data.** The task asked for "animated counters (structure only)" — built genuine Framer Motion count-up logic (`animate()` + `useMotionValue` + `useInView`, triggers once, respects `prefers-reduced-motion`), but no page passes it real numbers, because no school statistics are confirmed (same reasoning as `StatStrip` in Phase 1A — see that entry above). This is a different concern from `StatStrip`: `StatStrip` is the _specific_ homepage instance using qualitative highlights by design; `StatisticsGrid` is the _generic_ reusable primitive for whenever real numbers do exist. Documented the distinction explicitly in both READMEs so a future session doesn't try to delete one thinking it's a duplicate of the other.
- **`FAQAccordion` built without a headless UI library.** Needed real expand/collapse with correct ARIA (`aria-expanded`/`aria-controls`/`role="region"`), same category of problem as Phase 1A's `MobileNav`. Used a native `<button>` (keyboard support free) plus a CSS `grid-template-rows: 0fr → 1fr` transition (no JS height measurement, no Framer Motion needed for this one) — simpler and smaller than the equivalent with a dependency.
- **`BadgeGroup` and `Callout` semantic-color contrast.** First draft of `BadgeGroup` used `bg-{variant}/10 text-{variant}` (a light tint with the accent color as text) — caught before shipping that this fails contrast for `warning` specifically, since the `warning` token (`oklch(0.78 0.15 85)`, a light amber, chosen as a "state color visible against light chrome") is far too light to serve as _text_ against a near-white tinted background. Switched to solid `bg-{variant} text-{variant}-foreground`, using the token pairs exactly as designed (the `*-foreground` tokens exist specifically to guarantee contrast against their paired solid color). `Callout` avoided the same trap from the start by keeping body text in `foreground`/`muted-foreground` always, using the semantic color only for the icon and a decorative left border (non-text elements, held to a looser contrast bar).

**Manual verification — a real bug found and fixed:** Build, lint, typecheck, and format all passed clean throughout — as established in Phase 1A, that proves nothing about runtime correctness. Built a temporary preview route (`src/app/dev-preview-temp/`, outside any `_`-prefixed Next.js private folder so it would actually be routable — the first attempt used `_dev-preview` and silently produced a 404, since Next.js excludes underscore-prefixed folders from routing entirely) composing all 15 components with representative sample data, screenshotted it via headless Chrome + the DevTools Protocol at both 1440px desktop and 390px mobile (with proper `Emulation.setDeviceMetricsOverride({ mobile: true, ... })`, per the lesson recorded in the Phase 1A entry above), in dark mode (the environment's default). Found `PageHero`'s `variant="image"` treatment rendering white-on-white — no, black-on-black: title, subtitle, and secondary-CTA text were using `text-primary-foreground`, which is a _dark_ color in dark mode (correct for text on the light `primary` button, wrong for text on a permanently-dark photo overlay). Root cause identical in shape to the Phase 1A font-scoping bug: a token reused outside the context it was actually designed for. Fixed by switching every image-overlay text reference to literal `text-white` at appropriate opacities — deliberately _not_ a theme token, since the black scrim it sits on is itself not theme-dependent. Re-verified via a cropped screenshot of just that section post-fix, on both desktop and mobile. Also functionally clicked `FAQAccordion`'s first question via `Runtime.evaluate` + `.click()` and screenshotted the expanded state to confirm the accordion actually opens/closes, not just that it typechecks. Deleted the entire preview route, its temporary placeholder image, and reverted `next.config.ts`-adjacent experiments before finishing — nothing from the verification scaffold ships.

**Lessons learned:** The Phase 1A font bug and this phase's `PageHero` bug are the same _class_ of mistake — a semantic token borrowed for a purpose slightly outside its actual contract (`--font-sans` scoped to the wrong ancestor; `primary-foreground` used for "light text" instead of "text on the primary button"). ## 2026-07-17 — Phase 1B.1: Architecture Review & Refactoring

**Context:** A dedicated Staff-Architect-level review of the Marketing Section Library, one phase after it was built — not new feature work. The brief explicitly asked for the previous phase's own decisions to be re-derived from scratch rather than deferred to, component-by-component YAGNI challenge, a verdict on folder-ceremony proportionality, and a decision on the temporary-preview-page pattern used for manual verification.

**`components/` vs. `src/features/`, re-derived independently:** Scored the same question D-012 already answered — role-segmented `components/{ui,shared,website,admin,teacher}` vs. a `src/features/website/` migration — across 20 explicitly requested dimensions rather than assuming D-012's reasoning still held. Landed in the same place, but for a more complete reason than D-012 gave: most of the 20 dimensions (multi-tenancy, future mobile app, future SaaS, future API integration) turned out to be data-layer/routing concerns entirely orthogonal to the `components/` folder axis — a folder rename doesn't touch any of them. The dimension that actually mattered was **refactoring cost**: today, before Admin/Teacher exist and before any page imports these components, is the cheapest this migration will ever be — that's real and worth stating plainly rather than only defending the status quo. But migrating now means guessing at feature boundaries for two role folders that are still empty; better to add a concrete, testable trigger (see [DECISIONS.md § D-013](./DECISIONS.md#d-013--phase-1b1-architecture-review-reaffirmed-role-segmented-components-added-a-feature-subgrouping-threshold-rule)) than to either migrate speculatively or defer indefinitely with no re-evaluation condition — the latter was D-012's actual gap, not its conclusion.

**Component-by-component YAGNI challenge — what was found:** Read every one of the 15 Phase 1B components against "do we need this today, could this be delayed, is it over-engineered." Most held up: variant surfaces map 1:1 to real semantic tokens (`BadgeGroup`, `Callout`) or real, distinct visual needs already anticipated by the design brief (`ResponsiveImage`'s 4 aspect ratios, `PageHero`'s 4 variants — the latter explicitly speced by the original task, and legitimately the most complex component in the library because it will be the most reused once real pages exist). Two things stood out as genuinely worth flagging:

- **`SectionDivider`'s 4 variants were 2 variants wearing a costume.** `minimal` and `decorative` were invented during Phase 1B build-out with no spec and no consumer; inspecting the constants map showed both used the identical `border-border` color as `light` — the only real difference was DOM shape, meaning the `variant` prop silently conflated two unrelated concerns (tone vs. shape) under one name. Removed both, kept `light`/`dark` (the two variants that map to a real need — divider on `background` vs. on `surface-muted`). See [DECISIONS.md § D-014](./DECISIONS.md#d-014--sectiondivider-yagni-simplification).
- **`StatisticsGrid` is the weakest present-day justification in the library, but was left as-is.** It's the most Framer-Motion-heavy component built this phase (`useMotionValue`, `useTransform`, `useInView`, `animate()`) and has zero consumers and no real data — it's explicitly blocked in `TASKS.md` awaiting confirmed school statistics. Unlike `SectionDivider`, this wasn't invented scope — it was named explicitly in the original Phase 1B task ("animated counters — structure only"). Chose not to delete completed, tested, accessible code only to rebuild it later when data arrives (net negative), but recording the honest tension here: if this component had not already been explicitly speced, a stricter YAGNI read would have deferred building it until a real consumer existed. Worth remembering for future phases — an explicit spec is not automatically an exemption from "do we need this today."

**Folder-ceremony verdict:** Measured rather than guessed — the 15-component library is 1,071 lines of actual component/type/constant code spread across 71 files, with 666 lines of README prose (more than half the code volume) and 15 two-line `index.ts` barrel files. For a component like `ContentContainer` (17 lines of actual logic), the surrounding ceremony (README + types + index) outweighs the component itself several times over. Concluded the per-component-folder pattern itself is justified — it's genuinely what makes `PageHero` (144 lines, will be reused on every future content page) navigable, and `index.ts` barrel files are not ceremony despite their size — they're the entire mechanism that makes `sections/index.ts`'s central re-export and clean named imports work, and removing them would be a real DX regression, not a simplification. The actual ceremony problem was narrower: `README.md` was being written to a fixed six-section template regardless of whether all six sections applied, producing boilerplate like "Future Enhancements: None anticipated" on components too simple to need it. Documented a proportionality rule in [ARCHITECTURE.md § 6](./ARCHITECTURE.md#6-component-hierarchy) (omit sections that don't apply) rather than retroactively rewriting 15 already-shipped, already-reviewed READMEs for a stylistic economy that AI_RULES explicitly warns against chasing ("never rewrite approved code to improve it stylistically unless asked" — this task did ask, but only for genuine simplification, not wholesale rewriting of working documentation).

**Development playground — built, not just discussed:** Phase 1B's manual-verification step required hand-building a temporary preview route each time, including re-discovering mid-session that Next.js silently excludes underscore-prefixed folders from routing (the `_dev-preview` → `dev-preview-temp` rename from that phase). That cost is going to repeat every future phase unless something permanent replaces it. Built `src/app/dev/playground/page.tsx` — composes all 15 current components with sample data, guarded by `if (process.env.NODE_ENV === "production") notFound();`. Verified the guard is real, not aspirational: ran a clean production build (`rm -rf .next && NODE_ENV=production pnpm build`), started it (`pnpm start`), and confirmed `curl` against `/dev/playground` returns a genuine HTTP 404 — not just an unlinked route, an actually-404ing one. In development the same route returns 200 and was screenshotted via headless Chrome + CDP at 1440px in dark mode: no console errors, `PageHero`'s dark-overlay text fix from Phase 1B still holds, all 5 `BadgeGroup` semantic-token variants render with correct contrast, and `FAQAccordion`'s click behavior was verified end-to-end (`aria-expanded` `false` → `true` on click, confirmed via two separate `Runtime.evaluate` calls rather than reading the DOM synchronously right after `.click()`, which raced React's state update on the first attempt). Considered adopting Storybook instead, since `ARCHITECTURE.md` already flags it as a future consideration — rejected for this session specifically because it's a materially bigger decision (new dependency, new config surface, its own addon ecosystem) than this review's scope justifies; a zero-dependency guarded route solves the actual, immediate pain. See [DECISIONS.md § D-015](./DECISIONS.md#d-015--permanent-devplayground-route-replaces-temporary-preview-and-delete-pattern).

**Lessons learned:** Re-deriving a decision from scratch instead of citing the prior entry is worth doing even when the conclusion doesn't change — this pass surfaced the actual missing piece (a graduation threshold, and an honest acknowledgment of the refactoring-cost tradeoff) that citing D-012 alone would have left buried. Also: "challenge every component" is only a meaningful exercise if it's willing to find something — a review that reads all 15 components and concludes all 15 are perfect is indistinguishable from a review that didn't actually look closely enough to catch a naming smell like `SectionDivider`'s conflated `variant` prop.

Composition doesn't eliminate the need for the same real-browser verification discipline established in Phase 1A/1B; it just moves where the bug is likely to hide, from component internals to the specific data being passed in.

**Placeholder wording — reused existing precedent instead of inventing new phrasing:** `About/content.ts` already had `author: "[Principal's Name]"` and `role: "Principal, Pant Public School"` for the Principal's Message section, written in Phase 1C. Rather than inventing a differently-worded placeholder for `SCHOOL.principal.name` in the new config file (which would have silently changed the About page's rendered text the moment it was wired up), matched the exact existing wording, then had `About/content.ts` import it. This is the correct order of operations for a "replace duplicated literals, don't change visible output" refactor: let the already-shipped text win, make the config match it — not the reverse.

**Browser safety — same discipline as Milestone 5, no incidents:** Checked `ps aux` for the user's real Chrome and any leftover dev servers before touching anything (found the user's Chrome running normally, port 3000 free). Launched an isolated headless Chrome with a fresh `mktemp -d` profile on port 9444 (a new port, not reusing 9333 from the previous milestone, to rule out any chance of attaching to a stale session), captured its exact PID via `lsof`, and closed only that PID when done. Captured the dev server's PIDs immediately after start and killed only those three, rather than any `pkill` pattern. Verified the user's Chrome process count was unchanged (17, matching before and after) as a concrete check, not just an assumption.

**Manual verification — comparing against a known-good baseline, not just "no errors":** Because this was a pure refactor, the bar wasn't just "zero console errors" (necessary but insufficient) — it was "the rendered text is byte-for-byte what it was before." Pulled `headerBrand`/`footerBrand`/`footerLocation`/`title`/`h1`/`navLabels` via `Runtime.evaluate` from all four live pages and compared each string against what Milestone 5's own verification had already recorded as correct. Specifically re-checked the About page's Principal `<figcaption>` text and the Admissions Overview badges (`SCHOOL.classes`/`SCHOOL.affiliation`) since those were the two content-file changes with the highest risk of an accidental wording drift. All matched exactly. Also explicitly re-verified the theme-toggle `localStorage` key was still `"pps-theme"` post-unification, since that was the one genuine behavioral risk in this refactor (get the interpolated key wrong in the `<script>` template literal, and theme persistence breaks silently).

**A second pass caught one more real duplicate:** After the first refactor pass, re-grepped for the remaining literal strings and found `StatStrip.tsx`'s fourth stat item — `label: "Vidyadhar Nagar, Jaipur"` — was an exact match to `SCHOOL.locationShort` that the initial file-by-file pass (Hero, Footer, Header, MobileNav, layout, lib/seo) had missed, since `StatStrip` wasn't in the original "files containing 'Pant Public School'" grep (it only contains the location, not the name). Fixed it in the same session rather than treating the first pass as complete — worth noting because it's a reminder that grepping for one string (`"Pant Public School"`) doesn't surface every duplicated fact; the location needed its own separate check.

**Lessons learned:** A "replace duplicated literals" task has a real judgment call hiding in it at every site: is this an exact-match duplicate (replace it), a prose sentence that happens to contain the same words (leave it), or a single-use value that merely _could_ be centralized (leave it, note why). Treating all three the same — either force everything through config or centralize nothing — would have been the wrong call in both directions. The theme-storage-key bug found during the initial survey is the more important finding of this session: it's a real, if latent, defect that predates this refactor and would never have surfaced without deliberately grepping for duplication before writing any new code, exactly the "Survey" step the task's own Stage 1 asked for applied literally.

Documented the distinction explicitly (in this page's README and in `PROJECT_CONTEXT.md`) so a future reader doesn't mistake "Milestone 6B changed the nav" for a violation of 6A's constraint — they're different milestones with different rules, and conflating them would be a real documentation-accuracy failure.

**Browser safety — a genuinely different situation than Milestones 5/6A, handled without incident:** `ps aux` before touching anything found a `next-server` already running on port 3000 that this session had not started (different PIDs than anything closed at the end of the previous milestone). Also: when a task's safety rules are unusually explicit and specific, the right response is to extend that carefulness to adjacent situations the rules don't literally cover (the mystery dev server) rather than parse the rules narrowly and act freely everywhere they don't explicitly reach.

## 2026-07-17 — Content Readiness Framework (CRF)

**Context:** Build the onboarding framework used to track content readiness for every school on the platform, Pant Public School as first implementation — explicitly not a CMS, and explicitly required to be _generated_ from the four built pages' actual implementation, not manually invented.

**The derivation, done literally, not from memory:** Despite having written all four pages' `content.ts` files earlier this session, deliberately didn't rely on memory for the registries — ran `grep -n '\[.*\]'` against `src/config/school.ts`, `src/config/contact.ts`, and all four pages' `content.ts` files, and separately grepped for `ResponsiveImage`/`ImageText`/`<video` usage across every built page. Memory is exactly the kind of source that quietly drifts from what the code actually says, and this task's own explicit instruction ("do NOT manually invent registry entries — generate them from the implementation") was a direct warning against that failure mode. The grep output became the registry rows verbatim — 40 text items, 8 image items — each traceable to a specific file and line cited in its own "Replacement Notes" column.

**Two findings the naive version of this task would have missed:**

1. **Content vs. Process Note.** Four of the forty text placeholders aren't content at all — they're the `Callout` "Before this goes live" warnings engineering wrote _around_ unfinished sections (Admissions' document disclaimer, Academics' co-curricular and assessment disclaimers, Campus's gallery disclaimer). A mechanical placeholder-grep would have listed these as "content to collect from School Admin," which is wrong — School Admin doesn't write a warning about their own website; engineering removes it once the section it guards is resolved. Missing this distinction would have sent four nonsensical requests to a school ("please provide text for: 'this section shows planned categories only'"). Caught by actually reading what each bracketed string _was_, not just that it matched the placeholder pattern.
2. **Active vs. Latent vs. Silently-absent.** Not every `src/config/` placeholder is visible anywhere yet — `SCHOOL.address`, `.email`, `.phone`, `.academicSession`, `.motto`, and both `CONTACT.timings` fields are defined but consumed by zero built pages (verified by grepping every consumer, not assuming). These are real future requirements, but treating them with the same urgency as `IMG-005`–`008` (which literally render "[Photo pending]" on a live page today) would have been dishonest about what's actually at stake right now. A third bucket, "silently absent" (the Principal's portrait — `QuoteBlock` supports an optional `avatar` prop that simply isn't passed, with no visible defect), needed its own category too — conflating it with either of the other two would have either overstated urgency or missed it as a real enhancement opportunity.

**The empty registries are the clearest evidence of not inventing anything.** `DOCUMENT_REGISTRY.md` and `VIDEO_REGISTRY.md` came back genuinely empty — no page references a prospectus, downloadable form, or any video content anywhere. The tempting failure mode here isn't fabricating a _fact_ (this project already has strong discipline against that, per [CONTENT_GUIDELINES.md § 12](./CONTENT_GUIDELINES.md#12-what-this-platform-never-says)) — it's fabricating a plausible-sounding _requirement_ ("surely a school website needs a prospectus PDF," "surely a Principal's welcome video would be nice"). Both registries state explicitly what was searched and that nothing was found, rather than quietly padding them with generic school-website boilerplate to look more thorough. An empty, well-justified registry is a correct output of this framework, not an incomplete one.

**Priority is the one genuinely judgment-based layer, and it's flagged as such.** Every registry row's _existence_ is derived; each row's **P0/P1/P2** classification is a reasoned call layered on top (how visible, how central to trust, how embarrassing if seen live) — not itself extracted from the code. Distinguishing "derived fact" from "reasoned judgment applied to a derived fact" and saying so plainly (in `README.md`'s own Self Review section) matters here specifically because the task asked, directly, "did you invent anything?" — the honest answer is "not the requirements themselves, but the priority ranking is a judgment call, made transparently."

**Dashboard math is arithmetic on the registries, not a separate estimate.** Every number in `CONTENT_DASHBOARD.md` (48 total, 16 P0, the 13/7/14/3/11 per-page split, the 37/10/1 Active/Latent/Silently-absent split) is a row count or sum over the registry tables — re-derivable by anyone who counts the rows themselves. Deliberately included a "Structural Completeness" table alongside "Content Readiness" so the 0% readiness figure doesn't misrepresent the four pages as unfinished or broken — they're fully built and verified; only their _content_ is a known, tracked gap. Presenting only the 0% number without that context would have been technically true and practically misleading.

**Reusability was a design constraint, not an afterthought.** `SCHOOL_ONBOARDING_CHECKLIST.md`'s five phases (Before Development / Before QA / Before Launch / Content Collection / After Launch) name no school-specific fact in the checklist items themselves — only in the "status" notes appended under each phase. This mirrors the same discipline already applied to `PRODUCT_ARCHITECTURE.md`: the _process_ generalizes immediately; only _today's numbers_ are tenant-specific. A second school's onboarding reuses the identical five phases and produces its own registries via the identical grep-based derivation — verified by reading back through the checklist and confirming no phase heading or step description assumes Pant Public School's specifics.

**Lessons learned:** A "generate, don't invent" instruction is easy to satisfy shallowly (grep for brackets, dump them in a table) and much harder to satisfy honestly (also classify _what kind_ of placeholder each one is, distinguish _visible-today_ urgency from _latent_ future need, and be willing to ship two genuinely empty registries rather than padding them). The actual discipline this task tested wasn't "can you find every `[...]` in the codebase" — that part is mechanical — it was "can you resist adding anything beyond what the mechanical search actually found," which is the same instinct this project's YAGNI culture ([D-013](./DECISIONS.md#d-013--phase-1b1-architecture-review-reaffirmed-role-segmented-components-added-a-feature-subgrouping-threshold-rule), [D-014](./DECISIONS.md#d-014--sectiondivider-yagni-simplification)) has applied to code, now applied to a content registry instead.

## 2026-07-17 — School Domain Model

**Context:** Design the complete domain model for the School Operating System — the source of truth Epic B's future implementation will build against. Explicitly architecture-only: no Prisma models, migrations, API routes, or React components. Explicit instruction to think like a Solution Architect, not a frontend developer, and to design for the ~80% of independent Indian K-8 schools this platform is meant to eventually serve — not CBSE-only, not Rajasthan-only, despite Pant Public School being both.

**The single most consequential structural decision: `Enrollment`, not a mutable `Student.currentClass` field.** The obvious, easy version of this schema puts `class`/`section` directly on `Student` and updates them in place every promotion. That version breaks the moment a real report card or Transfer Certificate needs to say what class a student was in _two years ago_ — updating in place destroys that history. Modeling `Enrollment` as its own entity, one row per `(Student, AcademicYear)`, makes "what was this student's class last year" a normal query instead of a data-loss bug waiting to happen. Every other year-scoped entity (`AttendanceRecord`, `MarksRecord`, `ReportCard`, `PromotionRecord`, `TransferCertificate`) keys off `Enrollment`, not `Student`, directly — this is the one modeling choice worth a future engineer understanding before touching anything else in `docs/domain/`.

**Research had to precede design, not follow it — three real, non-obvious findings shaped the schema directly:**

1. **Board affiliation does not drive exam mechanics for Nursery–Class 8.** CBSE, ICSE, and State Boards all begin board-conducted examinations at Class 9/10 — for the age range this platform targets, affiliation mostly governs syllabus alignment and recognition status, not how a school runs its own Half-Yearly or Annual exams. Missing this would have tempted a `GradeScale` or `Examination` design that branches on `affiliationBoard`, which would be actively wrong for every school in this platform's target range, not just an unnecessary abstraction. `Examination`/`GradeScale`/`ExamTerm` are fully school-defined instead, with `affiliationBoard` staying a display/recognition fact.
2. **The RTE Amendment Act 2019 created real, active state-level divergence on promotion policy** — individual states can reintroduce Class 5/Class 8 detention with a re-examination provision, and Rajasthan (Pant Public School's own state) is one of the states that has. Hardcoding either the original national no-detention default or Rajasthan's own reintroduced-detention choice would each be wrong for a large fraction of the platform's eventual schools. `AcademicYear.promotionPolicy` is a JSON configuration blob specifically because this is genuinely per-state, not a fixed enum.
3. **UDISE+ is a near-universal compliance fact, not a Pant-Public-School-specific one.** Every recognized Indian school reports into it regardless of board or state, and a student's Permanent Education Number (PEN) persists across school transfers — which is why `udisePen` lives on `Student` (survives across `Enrollment`s) rather than being scoped to a single year's enrollment record.

**The Transfer Certificate got its own entity, not a generic `DocumentRecord` row, because its content is legally constrained, not editorial.** A TC's required field set — admission number, category, attendance, general conduct, promotion status, reason for leaving, fee dues — is close to identical across Indian states even though the printed _format_ varies; that's a genuinely different kind of "document" than a birth certificate scan, and modeling it as a flexible key-value document would have lost the ability to validate or query its required fields at all. `TransferCertificate` also intentionally has no update path beyond "issue a new dated one referencing the same student" — the same immutable-legal-record reasoning already applied to `ReportCard` (a generated snapshot, not a live view).

**PII classification surfaced a real compliance dimension the task didn't explicitly ask for, worth flagging rather than skipping.** Because every `Student` in this platform's scope is a minor, India's DPDP Act 2023 § 9 (verifiable parental consent, no behavioral tracking of children) applies to essentially the entire `Student`/`Enrollment`/`Attendance`/`Marks` surface. `DATA_DICTIONARY.md` introduced a "Minor PII" classification specifically so a future implementation doesn't have to rediscover this — and explicitly does _not_ claim to have designed a consent-capture mechanism, which is named as a real, flagged gap rather than silently absent.

**Future modules named as seams, not designed — a direct guardrail tension, resolved by precedent, not by guessing.** The task explicitly asked for "Future fee module integration," "Future transport integration," "Future parent portal integration," and "Future student portal integration" to be documented. [PROJECT_GUARDRAILS.md § 2](./PROJECT_GUARDRAILS.md#2-module-approval-process) requires the Module Approval Process before _any_ design work begins on an out-of-scope module. The resolution — naming only the attachment point (e.g., "a future `FeeInvoice` would reference `Enrollment`") without designing the module's fields, workflows, or rules — mirrors exactly how [PRODUCT_VISION.md § 9](./PRODUCT_VISION.md#9-future-expansion) already handles Parent/Student Portal at the product-vision level. This wasn't a new judgment call; it was recognizing an already-established pattern applied one level deeper.

**Entity count (~29) is deliberately larger than a minimal V1 slice, and that's stated explicitly rather than left implicit.** A task asking for "the complete domain model... source of truth for every future implementation" is a different kind of request than "design what Epic B's first milestone needs" — conflating the two would have either under-delivered against the actual ask or produced a schema that looks build-ready when it isn't supposed to authorize a build at all. [DOMAIN_MODEL.md § 11](./domain/DOMAIN_MODEL.md#11-recommended-build-order) exists specifically to prevent the completeness of the documentation from being mistaken for a mandate to build all 29 tables in one migration — the project's YAGNI discipline is preserved at _build_ time by being explicit about it now, not by under-designing the reference itself.

**Lessons learned:** "Design for 80% of Indian schools, not CBSE-only, not Rajasthan-only" is a testable constraint, not just a nice sentiment — it was applied by checking, per entity and per rule, "would this be wrong for a real school outside Pant Public School's own board/state," and pushing anything that failed that check into configuration. This is the same configuration-vs-code discipline [CONFIGURATION_GUIDE.md](./CONFIGURATION_GUIDE.md) already established for the website — applied here to a domain that has genuine _regulatory_ (not just stylistic) variation, which raises the cost of getting the split wrong from "looks inconsistent" to "silently produces incorrect compliance behavior for some future school." Also: architecture work done deliberately ahead of implementation is only valuable if it's honest about what it isn't — every document in `docs/domain/` states plainly that it authorizes no code, precisely so a future session doesn't mistake a thorough design reference for permission to start migrating.

Phone, email, and office hours moved to P0; address and visit hours to P1 — a judgment call, made transparently in the registry's own Replacement Notes column, following the same "priority is the one judgment-layered thing in this framework" discipline the CRF's own founding session established.

Also: "never invent contact details" is easy to satisfy for the obvious case (don't type in a fake phone number) and easy to violate for the non-obvious case (turning an honest placeholder string into a technically-functioning-but-semantically-broken `tel:` link) — the second kind of violation doesn't look like fabrication, it looks like normal engineering, which is exactly why it needs deliberate checking rather than trusting that "I didn't type in a fake number" is sufficient.

**Context:** Build the seventh public content page, `/documents` — admission/academic documents, mandatory public disclosures, school policies, a circulars-and-notices explanation, FAQ, and a contact CTA. Explicit instruction: "This is NOT simply a downloads page... the implementation must be generic enough to work for most Indian schools through configuration... This costs nothing extra today (it's still a committed TypeScript file, still Content per [CONFIGURATION_GUIDE.md § 2](./CONFIGURATION_GUIDE.md#2-what-belongs-in-content)) but means a future fifth category is one array entry, not a page rewrite — the literal, checkable meaning of "designed for configuration, not yet configuration-driven," rather than a vague gesture at future flexibility.

**Reused the Gallery Preview's "pending-asset card citing a registry ID" pattern for documents, not just photos — the same recognition that made `DataTable`'s promotion (Contact & Visit milestone) straightforward.** `Campus`, `School Life`, and `Contact` had all already established: a `FeatureGrid` card with a real title and a bracketed, registry-ID-citing description is the honest way to represent something that doesn't exist yet. The section's `Callout` states the category names plainly while explicitly deferring to `SCHOOL.affiliation` confirmation before anything is published — citing the actual unresolved config value directly in the disclaimer text, not just describing it abstractly, so a future reader can see exactly which fact is blocking.

**Circulars & Notices is the one section that isn't a placeholder list at all, and treating it like one would have been the easiest mistake to make.** The brief said "explain future capability, do not fabricate notices" — a first instinct might be to build a `FeatureGrid` of "sample" notice categories the way the document categories were built, each captioned "[Notice pending]." That would have been wrong: a document has a knowable, generic _type_ today (an Academic Calendar is a real category of thing regardless of its contents), but a notice's entire nature is that it's tied to a specific, unknowable future event — there's no generic "notice type" to name the way there's a generic "document type." **Browser safety, fifth consecutive round, no incident** — same pre-existing dev server on port 3000 left untouched, independent dev server and isolated Chrome instance on fresh ports, PID-targeted cleanup verified at the end.

**Lessons learned:** A brief that contains two instructions in apparent tension ("generic enough for configuration" + "don't build the configuration engine") usually isn't actually contradictory — it's asking for the _shape_ of the eventual solution to be visible now, without the _mechanism_ being built early. The array-plus-loop pattern here is a reusable answer to that shape of request, worth reaching for again if a future page's brief asks for the same kind of forward-compatibility without authorizing the infrastructure to back it. Also: "do not invent X" instructions are easiest to satisfy for the object nearest the instruction (don't invent a disclosure requirement) and easiest to violate for an adjacent object nobody explicitly named (don't invent a filename for that disclosure's PDF) — worth checking one level past the literal instruction each time, not just the literal instruction itself.

## 2026-07-18 — Physical Database Review

**Context:** Review the School Domain Model as a Database Architect thinking about a 15-year maintenance horizon, ahead of Epic B's actual Prisma implementation. Explicit framing: "Think like an engineer responsible for maintaining this system for the next 15 years. Not like someone trying to finish a sprint." Architecture only — no Prisma schema, SQL, or migration.

**The review's job was narrower than the domain model's, and staying inside that boundary was itself the discipline.** The domain model decided _what_ exists and _how it relates_; this review's only job was _how it's physically realized_ — primary keys, indexes, constraints, partitioning, migration order. Every temptation to second-guess an entity's existence or a relationship's direction was set aside explicitly, even where a physical concern (e.g., `Enrollment` being the FK target of three high-volume tables) touches conceptual territory — the four required questions in the task brief are answered by _restating and sharpening_ what `Enrollment`-as-hub already meant, not by re-deciding it.

**The single highest-value finding came from asking "what happens to this in 15 years," not "does this work today."** `AcademicYear.isCurrent` and `TeacherAssignment.isClassTeacher` were both correctly identified in the domain model as needing "exactly one per scope," and both were explicitly left as application-only invariants — a reasonable call for a conceptual document, but a database review exists specifically to ask "what happens the day a concurrent request, a retried mutation, or a future engineer's second code path violates this rule silently." Postgres partial unique indexes (`WHERE isCurrent = true`, `WHERE isClassTeacher = true`) turn both into physically impossible violations for the cost of one migration line each — the kind of fix that's nearly free before the first row exists and genuinely hard to retrofit once an application has shipped for months trusting an invariant that was never actually guaranteed.

**Growth projections turned "might need partitioning eventually" into a concrete, checkable number, which changed how confidently the recommendation could be stated.** Rather than asserting `AttendanceRecord` "will get large" as an abstract claim, worked the actual arithmetic from numbers already established elsewhere in this project's own docs (200–2,000 students, ~220 school days/year, both from `PRODUCT_VISION.md § 3`): 600,000–6,600,000 rows over 15 years for _one_ school, 10–300 million at a modest multi-school Epic H scale. This is the difference between a hand-wavy "consider partitioning someday" and a specific, defensible "partition from Migration 000, because here's the number that makes retrofitting it later expensive." The same exercise, applied to `Student`/`Guardian`/`Section`/`Subject`, produced the opposite, equally useful conclusion: these tables will never cross a few thousand rows even at 15-year scale, so no index/partitioning effort spent on them would ever pay for itself — worth stating explicitly rather than applying a uniform "big tables need big-table treatment" instinct everywhere.

**Found three real gaps in the domain model's own illustrative schema, and deliberately did not fix them in place.** `Section` and `Subject` were each missing a uniqueness constraint their own conceptual description implies, and `TransferCertificate.tcNumber` was drafted globally unique when it should be per-school — the last of which is a genuine, if latent, multi-tenancy bug: it would work perfectly for Pant Public School as the only tenant and then break the instant a second school's TC numbering collided with the first's. Correcting these felt tempting mid-review (they're small, obviously-right fixes), but a review's job is to review and flag, not to unilaterally edit the document under review while claiming to only be reading it — the same discipline this project already applies to its own decision-making layers (a review doesn't quietly become an implementation). Flagged explicitly for `DATABASE_SCHEMA.md`'s own next edit instead.

**Recommended against something, on purpose, to keep the review honest about scale.** Postgres Row-Level Security would be a genuinely elegant way to enforce `schoolId` scoping at the database layer — and was explicitly _not_ recommended for V1, because RLS's operational cost (session-variable plumbing, harder-to-reason-about migrations, less mature Prisma support) only pays for itself once `schoolId` is protecting genuinely different customers from each other, not organizing one customer's own data. Naming this as "considered and deferred" rather than either recommending it prematurely or omitting the consideration entirely matters here — a 15-year review that only ever says "add more infrastructure" isn't actually exercising judgment, it's pattern-matching to "more robust must be better."

**Lessons learned:** A physical-design review earns its keep specifically by asking questions a conceptual model has no reason to ask — "what does Postgres actually guarantee here, independent of application code," "what does this table's row count look like in year 12, not year 1," "which of these three plausible primary-key strategies actually matters, and for which specific tables." None of these are questions the domain model was wrong not to answer; they're a different altitude of question entirely, which is the whole justification for a dedicated review layer between conceptual design and Prisma implementation rather than deciding all of it in one pass.

## 2026-07-18 — Pre-Prisma Consistency Verification

**Lessons learned:** A consistency verification's value is proportional to how mechanically it's performed — diffing entity lists, grepping for specific claims across documents, checking a "flagged for later" gap against whether "later" has actually arrived — not how thoroughly the reviewer already believes they understand the material from having written it. Prior authorship of a document set is exactly the condition under which real inconsistencies are easiest to miss on a re-read, because a familiar document reads as correct from memory of intent rather than from what it actually says on the page.

Two models today doesn't warrant it; a single flat `schema.prisma` file stays correct until that threshold is actually crossed, and premature folder-splitting would have been inventing structure the guide itself says not to add yet. Similarly, `docs/ARCHITECTURE.md § 2` and `docs/PROJECT_CONTEXT.md § 7` both independently name `src/lib/db.ts` — a single file, not a folder — as the Prisma Client singleton's already-approved location; followed exactly, and the two new utility modules (`env.ts`, `db-utils.ts`) were added as flat sibling files in `lib/`, matching how `seo.ts`/`motion.ts`/`utils.ts` already live there, rather than inventing a `db/` subfolder that would have contradicted the documented structure for no real benefit at this scale.

**Prisma 7's actual config API is not what any of the architecture documents assumed, and validating the schema immediately (before writing anything downstream) is what caught it.** `docs/database/PRISMA_IMPLEMENTATION_GUIDE.md § 2` specified `url`/`directUrl` directly inside `schema.prisma`'s `datasource` block — this repo's installed Prisma version (`7.8.0`) rejects that syntax outright; the IDE's own diagnostics flagged it as an error the moment the file was written, before any migration was generated against it. Prisma 7's actual model splits connection configuration into `prisma.config.ts` (used by Migrate, reading `DIRECT_URL`) and a driver adapter (`@prisma/adapter-pg`) passed to `PrismaClient` at runtime (reading `DATABASE_URL`) — a real, unavoidable divergence from documentation written before this Prisma major version's release, not a stylistic choice to weigh. Running `prisma validate` immediately after every schema edit, rather than batching changes and validating once at the end, is what surfaced this early enough to fix cleanly instead of discovering it after migrations were already hand-written against the wrong syntax.

**A second real bug was found by actually running the code, not just typechecking it.** `prisma.config.ts`'s original `import "dotenv/config"` only loads `.env` — never `.env.local`, which is what `.env.example`'s own header comment (and Next.js's own convention, already relied on elsewhere in this app) instructs contributors to use for real local values. This wouldn't have shown up in `pnpm run typecheck`/`lint`/`build` — none of those actually invoke Prisma Migrate against real env vars — but did show up the moment a smoke test actually ran `checkDatabaseHealth()` and `prisma/seed.ts` end-to-end via `tsx`, first failing with a Zod validation error (`DATABASE_URL: undefined`) before the fix, then failing gracefully with `ECONNREFUSED` after it (the correct, expected failure mode given no live Postgres exists in this environment). The distinction matters: static checks confirm code _compiles_, not that it _runs correctly against its actual entry points_ — this bug lived exactly in that gap, and would have silently broken the first real `prisma migrate deploy` against a production database if left uncaught.

**Seed data required extending this project's "never fabricate a fact" discipline from public-site copy into database rows, for the first time.** `docs/database/PRISMA_IMPLEMENTATION_GUIDE.md § 5` says to seed one `School` and one `AcademicYear` "sourced from `src/config/school.ts`'s existing static values" — but `academicSession` there is still an unconfirmed bracketed placeholder string, not real dates, and `AcademicYear.startDate`/`endDate` are non-nullable `DateTime` columns that need an actual value to seed at all. The website's own established rule (`CONTENT_GUIDELINES.md § 12`: never fabricate a fact) doesn't translate literally to a database column — writing literal placeholder text like `"[Academic session — to be confirmed]"` into a `DateTime` field isn't possible, and writing it into a `String` field would be actively wrong in a way it isn't wrong in UI copy (a database row isn't read the same way a page is). The resolution: seed a computed April–March window using the well-known, real Indian academic-year convention (a structural fact about how Indian schools generally operate, not an invented fact about this specific school), paired with an explicitly-unconfigured `promotionPolicy` JSON marker rather than a fabricated real policy — both flagged in `D-027` and this sprint's own report as provisional, pending School Admin confirmation, the same traceability discipline already used for every bracketed placeholder on the public site.

**Lessons learned:** Architecture documents describe intent at the time they were written; a real Prisma version installed months later can silently diverge from that intent in ways only actually running the tooling reveals — `prisma validate`, a real `tsx` execution of the seed script, and a standalone health-check smoke test each caught a distinct, real bug that static analysis alone would have missed. None of the three bugs found here were architecture problems requiring a redesign — all three were implementation-detail corrections squarely within this sprint's own "genuine implementation discovery" allowance, applied narrowly (a doc-guide correction, a config fix, a seed-data judgment call) rather than treated as license to redesign anything the guide got wrong.

## 2026-07-18 — Sprint 1: Identity Foundation

**Context:** Directly continuing Sprint 0, with one changed condition stated upfront: "The PostgreSQL database now exists." Scope: Migration 002 — `User`, `Role`, `UserRole` (if justified), `Account`, `Session`, `VerificationToken` — plus a new `repositories/`/`services/`/`validators/` layer this task's own instructions specified, seeded with three named roles and zero users.

**The task named a `UserRole` model "if justified" — the right call was to not build it, and to build something the task didn't literally name instead.** A many-to-many join table between `User` and `Role` would be textbook over-engineering for a system where [D-001](./DECISIONS.md#d-001--three-roles-only-for-version-1) fixes exactly two authenticated permission tiers and every real user needs exactly one of them — building it anyway "because the task mentioned it" would have been exactly the kind of unjustified complexity [AI_RULES.md § 5](./AI_RULES.md#5-quality-bar) warns against. But the task's own seed list ("Administrator, Principal, Teacher") revealed a real tension a flat `role` enum field couldn't resolve: two of those three names need to grant the _same_ access level. The actual right design — a `Role` lookup table with a `name` (job title) separate from an `accessLevel` enum (the real permission tier) — wasn't literally requested by either the task or any existing document; it was derived from reading [D-001](./DECISIONS.md#d-001--three-roles-only-for-version-1) and [PROJECT_CONTEXT.md § 4](./PROJECT_CONTEXT.md#4-user-roles)'s "Admin: Principal / Management / School Office" phrasing together and noticing they only reconcile with the seed list if job title and permission tier are modeled as two different things. This is the sprint's clearest example of "architecture is frozen" not meaning "stop reasoning" — the frozen constraint (three roles, D-001) was exactly what made the naive reading of the task's own instructions wrong.

**Auth.js's Prisma adapter has a real, external, non-negotiable field-naming contract — and it collides with this project's own naming convention in exactly one place.** `DEVELOPMENT_CONVENTIONS.md § 1` says camelCase Prisma fields map to snake_case columns via `@map`. The `@auth/prisma-adapter` package, however, constructs its own Prisma `create`/`update` calls using literal keys straight from the OAuth2 token response shape — `access_token`, `refresh_token`, `expires_at`, `token_type`, `id_token`, `session_state` — and does not rename them. `@map` only renames the underlying SQL column, not the field name the Prisma Client (and therefore the adapter) actually writes to; declaring these fields as camelCase-with-`@map` would leave the adapter writing to field names that don't exist in the generated client, silently dropping every OAuth token on write. The fix was to keep these specific fields as the adapter's own literal names — a narrow, deliberately-documented exception, not a convention violation overlooked. This is the kind of bug that wouldn't surface in `prisma validate` or `tsc` at all — only in actually reading the adapter's own source behavior, which is exactly why it's called out with its own schema comment rather than trusted to be self-evident from the code.

**Two static-analysis-invisible bugs were caught by insisting on live verification against a real database, not settling for Sprint 0's schema-validate-only bar.** With Neon now reachable, this sprint ran `prisma migrate dev --create-only` for real (not the `migrate diff` workaround Sprint 0 needed) — and the generated migration folder's timestamp prefix (`20260717212444`, the machine's UTC clock) sorted _before_ Sprint 0's synthetic timestamps (`20260718...`), which would have made a fresh `prisma migrate deploy` try to apply Identity before School Foundation existed, failing immediately on the FK to `schools`. Renaming the folder fixed it — but only actually generating a real migration (not just diffing schema text) surfaced the problem at all. Separately, the repository/service pattern initially seemed complete after `tsc`/`eslint` passed clean — it wasn't confirmed correct until `createIdentityUser()` was actually run against the live database, verified to write both the `User` row and its `AuditLog` entry in one transaction, and cleaned up. Static checks confirm the code _compiles_; only execution confirms the transaction boundary and the FK relationships actually hold.

**Lessons learned:** A live database changes what "verified" means, not just what's possible — Sprint 0's schema-validation bar was the right bar for that sprint's actual constraint (no reachable Postgres), but this sprint had no excuse to settle for less once a real database existed, and two of this sprint's three real findings (the migration ordering bug, the transaction-boundary confirmation) only surfaced because the work was actually executed, not just statically checked. The `Role`/`accessLevel` design is this session's clearest example yet that "don't redesign architecture" and "reason carefully about what the architecture actually implies" are different instructions — the first was honored throughout; the second is what produced the one design decision this sprint that no single document had already made for me.

## 2026-07-18 — Sprint 2: Academic Foundation

**Context:** Directly continuing Sprint 1, with a new element: the task's own instructions now encode the prior architecture review's conclusions as "Permanent Architectural Decisions" (`roleId`, `accessLevel`, Credentials provider, database-backed sessions, Argon2id, `src/lib/validations` canonical, Repository → Service → DTO → API → UI layering) — not to re-derive, just to build on. Scope: Migration 003 — `SchoolClass`, `Section`, `Subject` — plus repositories for five entities (including two, `school` and `academicYear`, that predate any repository layer) and one orchestrating academic service.

**Following an explicit renaming instruction is still worth independently verifying, not just complying with.** The task said "use `SchoolClass` instead of `Class` everywhere" — a directive, not a request for judgment. But checking _why_ mattered anyway: `class` is a reserved word in JavaScript/TypeScript, and Prisma's generated client code for a model literally named `Class` is genuinely awkward (and in some generated contexts invalid) — a real technical justification independent of the instruction itself. Confirming this before writing the schema meant the resulting code comments could state a real reason, not just cite an order followed — the difference between "the task said so" and "the task said so, and here's why that's actually correct," which matters for whichever engineer reads this schema in two years without this conversation's context.

**"No direct Prisma outside repositories" is a rule that decays quietly if not periodically re-verified against the whole codebase, not just new code.** Sprint 0's `prisma/seed.ts` predated any repository layer and accessed `School`/`AcademicYear` directly — entirely reasonable at the time, since no repository existed yet to route through. By Sprint 2, with a repository layer now established and this sprint's own self-review explicitly re-stating "no direct Prisma outside repositories" as a checklist item, that old code became a latent gap: technically pre-existing, not newly introduced, but a grep-based verification would have failed against it regardless of when it was written. Retrofitting `school`/`academicYear` repositories and refactoring `seed.ts` to use them — rather than treating the rule as applying only to code written _after_ the rule existed — is what actually keeps a repository-boundary rule meaningful over multiple sprints instead of becoming a rule that only binds the newest code.

**The academic service's transaction boundary decision required reading `TRANSACTION_BOUNDARIES.md` for a case it doesn't explicitly cover, not just the cases it names.** That document's table lists specific named operations (admission confirmation, attendance submission, teacher onboarding, etc.) — "creating a class with its sections" isn't one of them. But § 1's _general_ principle ("any mutation that writes to more than one table must be wrapped in a single transaction... including the write and its own AuditLog entry") applies regardless of whether the specific operation made the table. Recognizing that an undocumented specific case still falls under an already-stated general rule — rather than treating the absence of a named row as license to skip the transaction — is the same kind of "apply the standing principle to new territory" judgment the `INDEXING_STRATEGY.md`-over-`DATABASE_SCHEMA.md` precedent (Sprint 0/1) already established for this project.

**Removing seed.ts's own `PrismaClient` construction, once the repository layer existed to replace it, was a small but real simplification worth making even though it wasn't explicitly requested.** Sprint 0/1's `seed.ts` needed its own locally-constructed client specifically to route around the ESM-hoisting/dotenv-ordering trap discovered in Sprint 0. Once every write now goes through repositories/services (which default to the shared `src/lib/db.ts` singleton), that local construction became redundant — one fewer connection pool opened per seed run, and one less place a future engineer could accidentally diverge from the singleton's configuration. This is the kind of cleanup that's easy to skip ("it already works") but cheap to make while already touching the file for an unrelated reason, and meaningfully reduces the seed script's own surface area.

**Lessons learned:** A rule stated once ("no direct Prisma outside repositories," "every multi-table write needs a transaction") doesn't automatically re-apply itself to code written before the rule existed, or to operations a reference document's table doesn't explicitly enumerate — each of those requires someone to actually go back and check, not assume compliance because no violation was flagged at the time. This sprint's two real findings (the `school`/`academicYear` repository gap, the transaction boundary for an unnamed operation) were both this same pattern: an established principle, correctly not violated by anything written _after_ it existed, quietly unchecked against what came _before_ it.

## 2026-07-19 — Sprint 3: Student Foundation

**Context:** Continuing Sprint 2, with a real new architectural layer required for the first time: "Services must return DTOs. Never expose raw Prisma models outside repositories." Scope: Migration 004 — `Student`, `Guardian`, `StudentGuardian`, `Enrollment` — three repositories, one service module, "lifecycle-oriented services, not CRUD-oriented services."

**The permanent decision's exact wording — "Enrollment is the aggregation root for all academic history" — was precise in a way that mattered for the service design, not just the schema.** It would have been easy to read this as "Enrollment is THE aggregate root, full stop" and build one `createStudentAndEnroll()` function bundling identity creation and enrollment together, since that's how `docs/domain/EVENT_MODEL.md`'s `AdmissionApplicationConfirmed` event actually works in the full system (Student + Enrollment, same transaction). But the decision says root "for all academic history" — a scoped claim — and `EVENT_MODEL.md`'s own catalog lists `StudentEnrolled` as a _separate_ event from student creation, with its own distinct payload. Reading the two together pointed at the actual right design: `registerStudent()` (identity, no year-scoping) and `enrollStudent()` (the year-scoped fact) as two independently callable lifecycle operations, not one. This mattered concretely for the seed script, too — re-enrolling a returning student next year should never require "re-registering" them, and a combined function would have made that awkward or impossible to express cleanly.

**Deciding where DTOs live was a real design choice, not a mechanical one — and the answer follows this project's own established "colocate until a second consumer justifies promotion" pattern.** The three DTO files (`student.dto.ts`, `guardian.dto.ts`, `enrollment.dto.ts`) live inside `src/services/student/`, not a new top-level `src/dtos/` folder, even though "Repository → Service → DTO → API → UI" reads like DTO could be its own peer layer. Nothing yet needs a Student DTO from outside the student service (no Teacher service exists that would need to expose a student summary, for instance), so a shared top-level location would be exactly the kind of speculative generalization this project's D-020/D-023 precedent already rejects — promote a shape to a shared location on its second real consumer, not preemptively. The `StudentDTO`'s exclusion of `udisePen` was a similarly deliberate, not mechanical, choice: nothing forced it out structurally, but a government-issued minor's ID has no business appearing in a general-purpose "list of students" shape when `BUSINESS_RULES.md § 8` (DPDP Act 2023) already names this exact category of field as needing deliberate handling — this is the DTO layer actually doing the job it exists for, not just satisfying a "return a plain object" checkbox.

**A transaction-connection-contention bug was found only because the seed script was actually run against the live database, with its console output read carefully — not by any of `typecheck`/`lint`/`prisma validate`, all of which passed cleanly on the buggy version.** `registerStudent()`'s first implementation resolved an existing `Guardian` by id from _inside_ the `db.$transaction()` callback, using the shared singleton rather than the transaction's own `tx` client — a read competing with an open transaction for the same connection pool. This produced a real Postgres/driver deprecation warning the moment the seed script's create path actually ran (not on the idempotent re-run, which never reaches that code at all — a second signal that only exercising the real path surfaces real bugs). The fix — resolve every existing-guardian reference _before_ opening the transaction, reuse the already-fetched object inside it — is a small change, but finding it required treating "the seed script printed its success messages" as necessary, not sufficient, evidence of correctness, and actually reading the full console output rather than just checking the exit code.

**Correcting `Student.admissionNumber`'s uniqueness scope was the third time this exact class of bug has been caught in this project — worth noting as a pattern, not just a one-off fix.** `docs/domain/DATABASE_SCHEMA.md`'s illustrative `Student` block showed `admissionNumber String @unique` (global); `docs/database/DATABASE_REVIEW.md § 2` already flagged this as wrong for the same reason `TransferCertificate.tcNumber` was wrong (D-025/D-026): a single-tenant illustrative schema's uniqueness constraints don't automatically generalize correctly once a second school exists. This is now a recognizable, recurring failure mode in how the original domain-model illustration was drafted (schoolId-scoped uniqueness isn't the default assumption baked into how those blocks were written), which is useful to know explicitly: any future migration touching a field DATABASE_REVIEW.md hasn't yet reviewed should be checked against this same question before assuming the illustrative doc's constraint is correct as written.

**Lessons learned:** A precisely-worded permanent decision ("aggregate root for all academic history," not "the aggregate root") is worth reading exactly, not paraphrased loosely, because the imprecise reading and the precise reading produce genuinely different service designs. And this sprint's second real bug (like Sprint 1's migration-ordering bug and transaction-boundary confirmation) was invisible to every static check and only found by actually executing the code against a live database and reading what it printed — the project's now-consistent pattern across three sprints is that live execution, not static analysis, is where the genuinely dangerous bugs get caught.

## 2026-07-19 — Sprint 4: Teacher Foundation

**Context:** Continuing Sprint 3, with a new explicit dependency-review requirement upfront ("before writing code... if any answer is NO, stop, explain, and justify") and a new explicit rule ("repositories NEVER call repositories") that this task's own instructions asked to be documented permanently if it wasn't already. Scope: Migration 005 — `Teacher`, `TeacherQualification`, `TeacherAssignment` — three repositories, one service module with a `dto/` subfolder, four lifecycle functions.

**The dependency-review requirement caught something worth catching, even though every answer came back "yes, additive."** Re-verifying `AttendanceSession`/`MarksRecord`'s illustrative schema directly (not from memory) confirmed both reference `User`, not `Teacher` — meaning the entire permission-check relationship between a teacher and what they're allowed to touch is an _application-layer_ concern (query `TeacherAssignment` to check authorization) rather than a _schema-layer_ dependency. This is a meaningfully different answer than "Teacher isn't referenced anywhere" would suggest — `TeacherAssignment` genuinely matters to those future features, just not as a foreign key. Stating this distinction precisely in the dependency review (rather than a flat "no dependency") is the kind of thing a review is supposed to produce that a rushed "looks fine, continue" wouldn't.

**Making an already-true rule explicit surfaced that it actually was already true — worth checking, not assuming.** Before writing `docs/engineering/ENGINEERING_PRINCIPLES.md`, every existing repository (Sprints 1-3) was grepped for cross-repository imports, specifically to confirm "repositories never call repositories" was describing real, already-consistent behavior rather than retroactively inventing a rule three sprints' worth of code happened to accidentally comply with. It was — a genuine confirmation, not an assumption, and worth stating that way in the document itself rather than presenting the rule as newly-invented.

**`registerTeacher()`'s design required reading `TRANSACTION_BOUNDARIES.md`'s "Teacher onboarding" row literally, then checking whether the permanent "repositories never call repositories" rule actually forbade what that row asks for.** The row explicitly names `User` + `Teacher` + `TeacherQualification` × N as one transaction — which means the _service_ needs to call `createUser()` (a different domain's repository) alongside `createTeacher()`/`createTeacherQualification()` (this domain's own). Read too literally, "repositories never call repositories" could be mistaken for "each service only touches its own domain's repositories" — which would have made this literally impossible to build as specified. The actual rule (repository-to-repository calls forbidden; service-to-repository calls across domains fine) had to be reasoned through precisely, not just pattern-matched from the rule's short name, and then stated precisely in the new documentation specifically so a future reader doesn't make the same over-broad misreading.

**A second, more benign cause of the same warning class Sprint 3 fixed was found by testing the hypothesis directly, not assuming it was the same bug recurring.** When the "client already executing a query" warning reappeared during this sprint's seed run, the instinct could have been "the Sprint 3 fix must have missed a spot in the new Teacher code" — but `--trace-warnings` stack traces pointed at Prisma's own query-interpreter, inside `PgTransaction.performIO`, not at any application-level read-before-transaction ordering mistake. Testing the hypothesis that this was a _different_ cause — by deliberately reproducing it against Sprint 3's already-shipped, previously-verified-correct `enrollStudent()` — confirmed it: the exact same warning, same stack trace, in code Sprint 4 never touched. This distinction mattered for scoping the fix correctly: fix it cheaply in Teacher's own new code (already being written), name it precisely as a _different_ root cause than Sprint 3's bug in the decision record, and explicitly decline to retroactively fix Sprint 3's code for it — since retroactively "fixing" already-shipped, correct-data-producing code for a cosmetic warning would have been scope creep this sprint's own "Teacher Foundation only" instruction doesn't authorize.

**`updateTeacherAssignment()`'s end-and-replace design came directly from taking `DATABASE_REVIEW.md § 7`'s own stated reasoning seriously, not from the task's example function names alone.** The task listed `updateTeacherAssignment()` as an example lifecycle function without specifying its semantics. `DATABASE_REVIEW.md § 7` already explains, for a completely different reason (soft-delete strategy), why a `TeacherAssignment` row's `sectionId`/`subjectId` should never be mutated once created: doing so would make "who was actually teaching this section on a given historical date" unanswerable, since `AttendanceRecord`/`MarksRecord` (future tables) would need to resolve that fact against whatever the row says _now_, not what it said _then_. Recognizing that this existing physical-design reasoning directly determines what "update" should mean here — end the old fact, start a new one, never overwrite — turned an underspecified example function name into a well-grounded design decision.

**Lessons learned:** A permanent, project-wide rule (this sprint's "repositories never call repositories," Sprint 3's "resolve reads before opening a transaction") is worth writing down with enough precision that a future reader can tell the difference between what it forbids and what it doesn't — a rule stated too tersely invites over-broad misapplication as easily as a rule left unstated invites drift. And a warning that looks like a bug already fixed once deserves the same "trace it, don't assume it" treatment as a genuinely new one — the fix for a recurring symptom isn't always the same fix, and confirming that (by testing against untouched, already-shipped code) is what separates an actual diagnosis from a guess that happened to also work.
