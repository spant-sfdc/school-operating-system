# Content Readiness Framework (CRF)

**This is not a CMS.** It doesn't store or serve content, and it doesn't replace `src/config/` or any page's `content.ts` as the actual source of truth the website reads from. It's the **onboarding framework** — the process and registries used to track what content a school needs to provide, in what state each piece is, and when a page is genuinely ready to go live. Pant Public School is its first real implementation, the same way it's the first real implementation of the platform itself ([PRODUCT_VISION.md](../PRODUCT_VISION.md), [D-019](../DECISIONS.md#d-019--product-pivot-school-operating-system-sos-pant-public-school-as-first-implementation)).

## How This Framework Was Built

Every entry in every registry in this folder was **generated by reading the actual implementation** — grepping `src/config/school.ts`, `src/config/contact.ts`, and all four built pages' `content.ts` files for bracketed placeholders (`[...]`), and checking every page's `page.tsx`/`sections.ts` for image/video component usage. Nothing here was written from memory, assumption, or a generic "what would a school website need" template. See each registry's own header for the specific search performed, and [IMPLEMENTATION_LOG.md](../IMPLEMENTATION_LOG.md)'s entry for this milestone for the full derivation narrative.

## The Status Lifecycle

Every content item in every registry moves through the same six stages, in order, never skipped:

```
Required → Requested → Received → Approved → Optimized → Published
```

- **Required** — identified as needed (this is where all 48 items stand today, at this framework's creation).
- **Requested** — an explicit ask has gone to School Admin (or, for the 4 Process Note items, an engineering task exists to remove it).
- **Received** — the content/asset has arrived, not yet reviewed.
- **Approved** — School Admin has confirmed it's accurate.
- **Optimized** — a technical pass (image compression/resizing, copy-editing) is done.
- **Published** — live on the actual page, replacing the placeholder.

## Two Kinds of Registry Item — Read This Before Using the Registries

Not every bracketed placeholder in the codebase needs the same kind of attention:

- **Content items** (36 of 40 text items, all 8 image items) — a real fact, story, or asset **School Admin** must supply.
- **Process Notes** (4 text items) — an internal `Callout` warning engineering placed in the page itself ("Before this goes live: …"). These are not collected from anyone; they're **removed by Engineering** once the content they guard is Approved. Treating them as content-to-collect would be a real mistake — see [TEXT_REGISTRY.md](./TEXT_REGISTRY.md)'s own note on this.

## Self Review

**Did you derive registries from the project, or invent them?** Derived — every ID in every registry traces to a specific file and line found by direct search, cited in that item's own row. Two registries ([DOCUMENT_REGISTRY.md](./DOCUMENT_REGISTRY.md), [VIDEO_REGISTRY.md](./VIDEO_REGISTRY.md)) came back empty because the search genuinely found nothing — they were left empty rather than padded with plausible-sounding future items, which would have been exactly the kind of invention this task explicitly ruled out.

**Did you invent anything?** The closest calls, stated honestly: (1) _Priority_ (P0/P1/P2) is a judgment applied on top of derived facts, not itself derived — reasoned from how visible/central each item is on its page, documented per-item so the reasoning is checkable, not asserted. (2) The Active/Latent/Silently-absent distinction in [IMAGE_REGISTRY.md](./IMAGE_REGISTRY.md) and [CONTENT_DASHBOARD.md](./CONTENT_DASHBOARD.md) required tracing which config fields are actually consumed by a component versus merely defined — real code-reading, not a guess. No content requirement, image slot, document, or video was added because it seemed like something a school website "should" have.

**Can future schools reuse this process?** Yes, and it was built that way deliberately: [SCHOOL_ONBOARDING_CHECKLIST.md](./SCHOOL_ONBOARDING_CHECKLIST.md)'s phase structure names no school-specific fact in its checklist items, only in its "status" notes. A second school's onboarding would run the identical derivation (grep its own `content.ts` files) and produce its own registries and dashboard — the _process_ is already platform-level; only _today's numbers_ are Pant-Public-School-specific. See [PRODUCT_VISION.md § 7](../PRODUCT_VISION.md#7-product-philosophy) for why that's the right order (generalize from one real proof, not speculation).

## Relationship to the Rest of the Documentation Set

This framework tracks _content_, not _code_ or _architecture_ — [CONFIGURATION_GUIDE.md](../CONFIGURATION_GUIDE.md) already drew the boundary between configuration, content, code, and future database records; this folder is entirely about the "content" (and the config-layer facts that happen to also be unconfirmed) side of that boundary, made concrete and trackable. It does not duplicate [TASKS.md](../TASKS.md) or [FEATURE_STATUS.md](../FEATURE_STATUS.md) — those track engineering work; this tracks content collection, a genuinely different workstream that runs in parallel.
